## Java特性




【问题】对 Java 平台的理解？

【参考答案】

1）**一次编译，到处运行（Write Once, Run Anywhere）**  
Java 源码（`.java`）先由 `javac` 编译成与平台无关的字节码文件（`.class`），这一步只做一次。  
在不同操作系统（Windows / Linux / macOS）上安装各自实现的 **JVM**，JVM 负责把同一份字节码翻译为各自平台上的机器码执行，因此达到了“同一份编译产物在多平台运行”的效果。

2）**JVM 在跨平台中的作用**  
- 每个平台有自己的 JVM 实现（HotSpot、OpenJ9 等，以及不同 OS 的变体），对上暴露统一的字节码指令集，对下屏蔽 CPU 指令集、操作系统差异。  
- 字节码在不同 JVM 上得到相同语义的执行结果，这是 Java 平台无关性的基础。

3）**编译 + 解释 / 即时编译的混合执行模式**  
- 第一步：`javac` 编译生成字节码（静态编译）  
- 第二步：JVM 通过 **解释执行器** 逐条解释字节码，或者通过 **JIT（Just-In-Time Compiler）** 把热点代码编译成本地机器码并做优化；Java 9 之后还支持 **AOT 编译器**，可以在部署阶段直接生成机器码。  
- 因此 Java 不是“纯解释型语言”，而是**先编译成字节码、再由解释 + JIT/AOT 混合执行**的语言。

4）**Java 平台的整体构成**  
- **语言本身**：语法、类型系统、OOP 特性等；  
- **JVM**：虚拟机规范 + 各种具体实现（HotSpot 等）；  
- **标准类库（Java SE API）**：集合、并发、IO/NIO、网络、安全等；  
- **JDK 生态**：编译器、调试/诊断工具、构建工具、各类框架。

【延伸考点】
- JVM 规范 vs JVM 实现（HotSpot 只是实现之一）。  
- 字节码结构、类文件格式（常量池、方法表、属性表等）。  
- JIT 优化手段：内联、逃逸分析、锁消除、栈上分配等。  
- AOT 与 JIT 的优缺点，对启动速度与峰值性能的影响。





【问题】Java 的特性是什么？Java 是解析运行的吗？

【参考答案】

1）**语言层面的特性**
- **面向对象**：封装、继承、多态，支持接口、抽象类、内部类等。  
- **类型安全**：强类型、泛型（Java 5+）、受检异常机制。  
- **函数式特性**：Lambda 表达式、方法引用、Stream API（Java 8+）。  
- **并发支持**：`java.util.concurrent` 包，线程、线程池、锁、原子类等。

2）**平台层面的特性**
- **平台无关性**：依赖 JVM 执行字节码，实现“编译一次，到处运行”。  
- **自动内存管理**：GC（垃圾收集），开发者不需要手工 `free` 内存。  
- **安全性**：类加载机制、字节码校验、安全管理器（SecurityManager，已逐步废弃）。

3）**类库与工具链**
- **丰富标准类库**：集合、并发、IO/NIO、网络、反射、正则、安全等。  
- **JRE**：Java 运行环境，包括 JVM + 标准类库。  
- **JDK**：在 JRE 基础上，增加 `javac`、`javadoc`、`jdb`、`jps`、`jstack` 等开发与诊断工具。

4）**Java 的执行模式是否是“解析运行”？**
- Java 源代码先由 `javac` **编译**为 `.class` 字节码。  
- JVM 在运行时对字节码采取 **解释执行 + JIT 编译** 的混合模式：  
  - 解释执行：字节码逐条由解释器翻译为对应平台指令执行；  
  - JIT 编译：对**热点代码**进行本地编译并优化，提高运行效率；  
  - Java 9 引入 **AOT 编译器**，可在部署期直接生成机器码以加速启动。  
- 因此：**Java 不是简单“解析运行”的脚本语言，而是“先编译成字节码，再由 JVM 解释 / JIT / AOT 执行”的语言。**

【延伸考点】
- Java SE / EE / ME 的区别与历史演进。  
- 不同 GC 算法（Serial、Parallel、CMS、G1、ZGC、Shenandoah）及适用场景。  
- 即时编译器 C1 / C2、分层编译的概念。  
- 与其他平台（.NET CLR、Node.js V8）执行模型的对比。



【问题】1. 什么是 Java 虚拟机（JVM）？为什么 Java 被称作“平台无关的编程语言”？

【参考答案】

1）**JVM 的定义**  
Java 虚拟机（Java Virtual Machine）是一个**抽象的计算机模型**，能够加载并执行符合 JVM 规范的字节码（不仅限于 Java，可来自 Kotlin、Scala 等）。  
从实现角度看，它是一个本地进程，内部包含类加载子系统、运行时数据区（堆、栈、方法区等）、执行引擎（解释器、JIT）、本地方法接口等。

2）**平台无关性的原因**
- 开发者只需面向 **统一的 Java 平台（字节码 + API）** 开发，不必关心具体操作系统和 CPU。  
- 不同平台有不同的 JVM 实现（如 Windows / Linux / macOS 上的 HotSpot），但都遵循统一的 **JVM 规范**，保证相同字节码在不同平台得到相同的行为。  
- 因此，Java 被称为“**编写一次，到处运行**”的、**平台无关的编程语言**。

【延伸考点】
- JVM 的主要组成：类加载子系统、运行时数据区、执行引擎、本地接口等。  
- HotSpot、OpenJ9、GraalVM 等不同 JVM 实现的特点。  
- Java 还能通过 GraalVM Native Image 等方式生成本地可执行文件。



【问题】2. JDK 和 JRE 的区别是什么？

【参考答案】

- **JRE（Java Runtime Environment，Java 运行时环境）**  
  - 面向“运行”场景，包含：  
    - JVM 实现；  
    - Java 标准类库（`rt.jar` 等，JDK 9+ 为模块化 `java.base` 等）。  
  - 只提供运行 Java 程序所必需的组件，不包含编译器等开发工具。

- **JDK（Java Development Kit，Java 开发工具包）**  
  - 面向“开发 + 运行”场景，**包含 JRE**，并额外提供：  
    - `javac` 编译器；  
    - `javadoc`、`jar` 等构建工具；  
    - `jdb`、`jps`、`jstack`、`jmap`、`jstat` 等诊断调试工具。  
  - 是开发人员安装的标准套件。

【注意】
- 从 **JDK 11** 开始，Oracle 不再单独提供 JRE 安装包，但“运行时环境”这一概念仍然存在。  

【延伸考点】
- JDK 8 与 JDK 9 模块化（Jigsaw 项目）后的目录结构差异。  
- 常用 JDK 命令行工具的使用场景。  
- JDK 发行版（Oracle JDK、OpenJDK、各云厂商定制版）的差异。




## OOP面向对象

### 封装继承多态抽象

- OOP面向对象编程


【问题】Java 是一个支持并发、基于类和面向对象的编程语言。下面列出了面向对象软件开发的一些优点：

【参考答案】

- **模块化**：通过类、包、模块，将复杂系统拆分为职责单一、边界清晰的模块，利于维护和扩展。  
- **复用性**：通过继承、组合、接口复用已有代码，减少重复开发。  
- **可维护性与可扩展性**：良好的封装与抽象，使得修改实现时对外部影响最小（开闭原则）。  
- **可读性与可理解性**：贴近业务概念建模（领域模型），让代码更易理解与沟通。  
- **多态性**：通过接口/父类引用指向子类实例，提高系统的灵活性与可扩展性。

【延伸考点】
- 面向对象设计原则：SOLID（单一职责、开闭、里氏替换、接口隔离、依赖倒置）。  
- 面向对象 vs 面向过程、函数式编程在 Java 中的结合（Lambda、Stream）。  
- 常见设计模式（工厂、单例、策略、模板方法、观察者等）与 OOP 的关系。



【问题】使用封装的一些好处？

【参考答案】

- **隐藏实现细节**：通过 `private` 字段 + `public`/`protected` 方法，只暴露必要接口，保护内部状态不被随意篡改。  
- **提高可维护性**：外部只依赖“对外接口”，内部实现可自由重构，而不影响调用方。  
- **增强安全性与一致性**：可以在 setter / 方法中做校验、约束，保证对象始终处于合法状态。  
- **提升模块化程度**：不同对象之间通过接口协作，避免直接操作彼此内部数据，降低耦合。

【延伸考点】
- 访问控制修饰符：`public` / `protected` / 默认（包可见） / `private` 的使用场景。  
- DTO / VO / Entity / Domain Model 等概念与封装的关系。  
- 不可变对象（Immutable Object）的设计与好处。



【问题】抽象和封装的不同点？

【参考答案】

- **抽象（Abstraction）**：  
  - 关注“**做什么**”，强调对共性行为和属性的抽取，忽略无关细节；  
  - 体现形式：抽象类、接口、方法签名、领域模型等。

- **封装（Encapsulation）**：  
  - 关注“**怎么做**”，强调把数据和行为打包在一起，并对外隐藏内部实现细节；  
  - 体现形式：访问控制符、getter/setter、对外暴露有限接口等。

- 二者关系：  
  - 抽象定义“对外可见的能力（接口）”；  
  - 封装通过隐藏内部状态与实现细节，**支撑**并实现这一抽象。

【延伸考点】
- 接口 vs 抽象类的区别与使用场景。  
- 如何通过接口抽象 + 封装实现“面向接口编程，而不是面向实现编程”。  
- 在重构中“提取接口 / 抽象类”的实践。



【问题】new 一个对象的过程和 clone 一个对象的区别？

【参考答案】

1）**通过 new 创建对象**
- 根据 `new` 后的类型信息，JVM 在堆上分配足够的内存空间；  
- 初始化对象头（Mark Word、类型指针等）；  
- 所有字段赋默认值（0、`false`、`null` 等）；  
- 按继承层次从上到下执行构造方法（包括实例初始化块），完成显式初始化；  
- 返回对象引用。

2）**通过 clone 创建对象**
- `Object.clone()` 是一个 **本地方法**，通常会：  
  - 在堆上分配与原对象同大小的内存；  
  - 直接按位拷贝原对象的字段值到新对象（**浅拷贝**）；  
  - 不会调用构造方法。  
- 要想让类支持 `clone()`，一般需要：  
  - 实现 `Cloneable` 接口（否则会抛 `CloneNotSupportedException`）；  
  - 重写 `clone()` 方法，并调整可见性（`protected` → `public`），必要时手动实现深拷贝。

3）**差异小结**
- `new`：调用构造器，遵循初始化逻辑，适合正常创建对象。  
- `clone`：跳过构造器，通过字段拷贝得到一个现有对象的“副本”，默认是浅拷贝，适合对已有状态进行复制。

【延伸考点】
- 浅拷贝 vs 深拷贝的区别与实现方式（拷贝引用 vs 拷贝对象）。  
- 更推荐的复制方式：拷贝构造器、静态工厂方法、序列化反序列化等，而不是滥用 `clone()`。  
- 为什么《Effective Java》不推荐使用 `Cloneable`。



【问题】为什么 Java 是单继承，但却是多实现的呢？

【参考答案】

1）**为什么类只支持单继承？**
- 多继承会带来经典的“菱形继承问题”，如多个父类提供同名字段 / 方法时，子类继承后会产生二义性：  
  - `class C extends A, B` 时调用 `c.print()`，不知道应该选择哪个父类实现；  
  - 增加类型层次的复杂度，影响可读性、可维护性和编译器实现难度。  
- 单继承 + 组合 能够覆盖绝大多数需求，同时避免多继承的复杂度，因此 Java 设计时选择了**类的单继承模型**。

2）**为什么接口可以多实现？**
- 接口主要用于定义“能力 / 行为契约”，不包含状态（在 Java 8 之前甚至没有默认实现）；  
- 即便多个接口中存在同名抽象方法，**实现类只需提供一个实现即可**，不会产生二义性问题；  
- Java 8 之后接口新增 `default` 方法，如果多个接口的 `default` 方法冲突，编译器会强制实现类显式重写并指定实现，依然可消歧。

3）**单继承 + 多实现的综合好处**
- 保留继承的表达能力与代码复用能力；  
- 通过多实现接口获得“多类型（多角色）”的能力（一个对象可以被视为多个接口类型）；  
- 避免 C++ 式复杂多继承带来的语义歧义与实现复杂度。

【延伸考点】
- 组合优于继承的设计思想。  
- Java 8 接口的 `default` 和 `static` 方法、多继承冲突的解决规则。  
- 与 C++ 多继承、C# 接口多实现的对比。






### 对象结构


【问题】对象的内存结构是怎样的？（以 HotSpot JVM 为例）

【参考答案】

在 HotSpot 中，一个普通 Java 对象在堆上的布局一般由三部分组成：

1）**对象头（Object Header）**
- **Mark Word**：存储对象运行时元数据，如哈希码、GC 分代年龄、锁标志位、偏向锁线程 ID 等；  
- **Klass Pointer（类型指针）**：指向对象对应的类元数据（Klass），用于在运行时知道“我是哪个类”；  
- 在 64 位虚拟机上，如果开启了 **压缩指针（CompressedOops / CompressedClassPointers）**，对象头大小和指针宽度会相应变化。

2）**实例数据（Instance Data）**
- 即类中声明的各个字段，包括从父类继承而来的字段；  
- JVM 会按照一定的规则对字段进行重排（如按对齐规则和类型宽度排序）以节省内存，但 **这一重排与 Java 内存模型中的“指令重排序”是两个概念，不要混淆**。

3）**对齐填充（Padding）**
- 为了满足对象起始地址必须是某个字节对齐（通常 8 字节对齐）的要求，JVM 会在必要时在对象末尾填充若干无意义的字节；  
- 这些对齐字节只为满足硬件和 JVM 的对齐要求，不存放业务数据。

【延伸考点】
- `-XX:+UseCompressedOops`、`-XX:+UseCompressedClassPointers` 对内存占用的影响。  
- 数组对象的内存布局（对象头 + 数组长度 + 元素）。  
- `Object.hashCode()` 的计算与 Mark Word 的关系；对象上锁（偏向锁、轻量级锁、重量级锁）时 Mark Word 的变化。  
- 如何通过 `jol-core` 等工具查看对象实际内存布局。

## java问题



### 标识符


【问题】如何判断一个标识符是否合法？

【参考答案】

1）基本规则（Java 语言规范）
- 标识符可以由：**字母、数字、下划线 `_`、美元符号 `$` 以及其他 Unicode 字符**（如中文）组成；  
- **不能以数字开头**；  
- 不能包含空格及如 `+ - * / % .` 等其他符号；  
- **不能与 Java 关键字或保留字同名**（如 `class`、`if`、`switch` 等）；  
- 大小写敏感，例如 `name` 与 `Name` 是两个不同的标识符。

2）实践中的命名规范（不是语法强制，但面试常问）
- 包名使用全小写，通常为反域名形式，如 `com.example.project`;  
- 类名、接口名使用大驼峰，如 `UserService`, `OrderController`;  
- 方法名、变量名使用小驼峰，如 `getUserById`、`userName`;  
- 常量名全部大写，单词间用下划线分隔，如 `MAX_VALUE`。

【特别注意】
- 从 Java 9 开始，**单独的下划线 `_` 不能作为标识符**，被保留为关键字，使用会编译报错。  

【延伸考点】
- `enum`、模块名（Java 9+）等在命名上的约定。  
- 为什么不推荐在业务代码中使用 `$` 作为标识符（IDE / 编译器生成代码会使用）。




### char和string


【问题】字符型常量和字符串常量的区别？

【参考答案（以 Java 为主）】

1）形式上的区别
- **字符常量**：单引号括起来的单个“UTF-16 代码单元”，如 `'a'`、`'中'`；  
- **字符串常量**：双引号括起来的 0 个或多个字符序列，如 `"hello"`、`"中国"`、`""`。

2）含义上的区别
- `char` 在 Java 中是一个 **16 位无符号整数类型**，表示一个 UTF-16 code unit，本质上是一个数值，可以参与算术运算；  
- `String` 是一个 **引用类型**，表示字符序列对象，变量中存放的是引用（指针），实际内容存放在堆中，并且具有不可变性（immutable）。

3）内存上的区别
- 一个 `char` 占 **2 字节**（16 bit）；  
- 一个 `String` 对象包含对象头、`coder`（Java 9+）、`byte[]` 或 `char[]` 等结构，长度随内容变化，并且字符串字面量会放入**字符串常量池**中以便复用。

【延伸考点】
- Java 中 `char` 不能完整表示所有 Unicode 字符（需要代理对 / 补充字符）。  
- `String` 的不可变性带来的线程安全、常量池复用、作为 Map key 的优势。  
- JDK 9 Compact Strings 改变了字符串底层存储结构（`char[]` → `byte[] + coder`）。



【问题】17、是否可以继承 String 类？为什么？

【参考答案】

- `String` 被声明为 `final` 类：`public final class String implements Serializable, Comparable<String>, CharSequence`，因此 **不能被继承**。  
- `String` 内部使用 `final` 数组（JDK 8 为 `char[] value`，JDK 9+ 为 `byte[] value + byte coder`）来存储内容，并通过设计保证其**不可变性（immutable）**。  
- 不可变带来的好处包括：  
  - 线程安全，可在多线程间安全共享；  
  - 可作为 Map 的 key，`hashCode()` 可缓存且稳定；  
  - 便于实现字符串常量池（String Pool），提升内存利用率与比较效率；  
  - 提高安全性（如类加载、反射、ClassName、文件路径、URL 等）。

【问题】17、字符串拼接原理？

【参考答案】

1）**运行时使用 `StringBuilder`/`StringBuffer`**
- 对于非编译期常量的 `+` 拼接，编译器会在字节码中生成类似代码：  
  - 创建 `StringBuilder sb = new StringBuilder();`（或以第一个字符串作为初始值）；  
  - 顺序调用 `sb.append(...)`；  
  - 最后调用 `sb.toString()` 生成新的 `String` 对象。  
- 因为 `String` 不可变，多次 `+` 拼接如果不使用 `StringBuilder`，会产生大量中间对象，性能较差。

2）**编译期常量折叠**
- 对于纯常量表达式，如 `"a" + "b" + "c"`，在编译期会被直接折叠为 `"abc"`，不会产生运行时开销。

3）**容量与扩容**
- `StringBuilder` 默认容量为 16，使用带初始字符串的构造器时，容量为 `str.length() + 16`；  
- 当容量不足时会扩容，通常按 `newCapacity = oldCapacity * 2 + 2` 规则扩展，并复制原数组，会有一定性能开销。

【延伸考点】
- 在循环中使用 `+` 拼接字符串的性能问题与规范建议。  
- 使用 `StringBuilder` vs `StringBuffer` 的区别与使用场景。  
- JDK 各版本下字符串拼接对应的字节码变化（如 `invokedynamic` 与 `StringConcatFactory`）。



【问题】19、String 和 StringBuilder、StringBuffer 的区别？默认容量是多少？

【参考答案】

1）**可变性与线程安全**
- `String`：不可变类，一旦创建内容不可修改，任何“修改”操作都会生成新的 `String` 对象；  
- `StringBuilder`：可变字符序列，方法**未加锁**，非线程安全，单线程场景下性能优；  
- `StringBuffer`：可变字符序列，大部分方法使用 `synchronized` 修饰，线程安全，相比 `StringBuilder` 有额外同步开销。

2）**适用场景**
- `String`：少量字符串拼接、字符串常量、方法参数 / 返回值、Map key 等；  
- `StringBuilder`：单线程场景下大量字符串拼接，如循环中构造 SQL、日志等；  
- `StringBuffer`：早期多线程环境常用，现在更多使用 `StringBuilder` + 线程封闭 或 其他并发容器。

3）**默认容量**
- 无参构造的 `StringBuilder` / `StringBuffer` 默认容量为 **16 个字符**；  
- 使用带字符串参数的构造器时，初始容量为 `str.length() + 16`；  
- 扩容规则：当追加字符导致长度超过当前容量时，会进行扩容，一般为 `newCapacity = oldCapacity * 2 + 2`，并伴随数组拷贝。

【延伸考点】
- 为什么 `String` 设计为不可变，以及对哈希、常量池、多线程的影响。  
- 日志框架（如 SLF4J）推荐的占位符方式与字符串拼接性能问题。  
- JDK 9 引入 `StringConcatFactory` 后，直接使用 `+` 拼接在某些场景下也能获得较好性能。



【问题】Java 9 对 String 的改动有哪些？

【参考答案】

- 引入 **Compact Strings**：  
  - JDK 8 及之前，`String` 内部使用 `char[]` 存储字符，每个字符固定占 2 字节；  
  - 从 JDK 9 开始，`String` 改为使用 `byte[] value + byte coder`：  
    - 如果内容可以用 `ISO-8859-1`（Latin-1）编码表示，则使用单字节编码（`coder = 0`），大幅节省内存；  
    - 否则使用 UTF-16（`coder = 1`）；  
  - 对相关类如 `StringBuilder`、`StringBuffer` 等也做了相应调整。

- 为保证兼容与性能：  
  - JVM 对大量字符串相关方法做了 Intrinsic（内联 / 本地）优化；  
  - 从上层 API 看，`String` 行为与之前版本保持一致，对业务代码透明。

【延伸考点】
- JDK 9 之后 `String` 内部实现变化对内存占用的实际收益。  
- JDK 9+ 还对 `String` 引入的一些性能相关改进（如 `StringUTF16`、`StringLatin1` 内部类）。  
- 通过 `-XX:-CompactStrings` 关闭 Compact Strings 的影响（了解即可）。



【问题】22、char 型变量中能不能存储一个中文汉字，为什么？

【参考答案】

- 在 Java 中，`char` 是一个 **16 位无符号整数**，表示一个 UTF-16 代码单元；  
- 对于大多数常见汉字，单个汉字可以用一个 UTF-16 代码单元表示，因此可以存入一个 `char` 变量，例如：`char c = '中';`；  
- Java 使用 Unicode 进行字符表示（逻辑层面），`char` 则存储其 UTF-16 编码值。

【注意】
- 对于某些超出基本多文种平面（BMP，U+0000~U+FFFF）的 Unicode 字符，需要使用 **一对 `char`（代理对）** 才能表示，此时一个 `char` 不足以表示一个完整“字符”的概念。  

【延伸考点】
- `Character` 类中对代码点（code point）的相关 API，如 `codePointAt`、`isSupplementaryCodePoint`。  
- 字符集（Charset）与字符编码（Encoding）的区别，以及 UTF-8、UTF-16 的不同。



【问题】如何将 byte[] 转为 String？需要注意什么？

【参考答案】

- 可以使用 `String` 提供的构造器：  
  - `new String(byte[])`  
  - `new String(byte[], Charset)` 或 `new String(byte[], String charsetName)`  
- 实际项目中应**显式指定字符集**，例如：  
  - `new String(bytes, StandardCharsets.UTF_8)`  
  - 避免使用无参构造 `new String(bytes)`，因为它会使用**平台默认编码**，可能与原始编码不一致，导致中文乱码等问题。

【延伸考点】
- 常见字符集：`UTF-8`、`GBK`、`ISO-8859-1` 等的差别。  
- `String#getBytes(Charset)` 与 `new String(byte[], Charset)` 的对应关系。  
- Web / 数据库 / 消息中间件等系统之间字符集不一致导致的编码问题排查思路。



【问题】什么是字符串常量池（String Pool）？

【参考答案】

1）基本概念
- 字符串常量池是 JVM 中专门用来**存放字符串常量和已被驻留（intern）的字符串对象引用**的区域；  
- 之所以可以实现字符串常量池，是因为 `String` 是不可变的（immutable），同一个字符串内容可以安全地在多个地方共享。

2）字符串的创建方式与常量池
- 使用双引号字面量，如 `String s = "abc";`：  
  - 编译期会在常量池中记录 `"abc"`，类加载时会在运行时常量池 / 字符串池中创建或复用该字符串；  
  - 多处使用 `"abc"` 会共享同一池中对象引用。  
- 使用 `new String("abc")`：  
  - 一般会在堆上再创建一个新的 `String` 对象，其内容与池中 `"abc"` 相同，但不是同一个引用。  

3）JDK 版本差异
- JDK 6 及之前：字符串常量池位于 **永久代（PermGen）**；  
- JDK 7 及之后：字符串常量池移动到 **Java 堆** 中，便于垃圾回收，提高灵活性。

4）`intern()` 方法
- `s.intern()` 会在字符串池中查找与 `s` 内容相同的字符串：  
  - 若已存在，返回池中字符串的引用；  
  - 若不存在，则将当前字符串加入池中（具体行为在不同 JDK 版本有细微差异），并返回池中引用。  

【延伸考点】
- 字符串常量池与享元模式（Flyweight Pattern）的关系。  
- 大量调用 `intern()` 对内存与 GC 的影响。  
- 字符串常量池与类文件常量池（Class 常量池、运行时常量池）的区别与联系。



【问题】你对 String 对象的 intern() 熟悉么？

【参考答案】

1）`intern()` 的基本行为  
- `s.intern()` 会在字符串常量池中查找“内容与 `s` 相同”的字符串：  
  - 如果已存在，则返回池中字符串的引用；  
  - 如果不存在，则将一个等值字符串放入常量池并返回池中字符串的引用。  

2）与字面量、`new String()` 的关系示例

String s1 = "aa";
String s2 = new String("aa");
String s3 = s2.intern();

System.out.println(s1 == s2); // false：一个在常量池，一个是堆中新对象
System.out.println(s1 == s3); // true：intern() 返回的是常量池中的引用

3）JDK 版本差异（了解）  
- JDK 6：常量池在永久代（PermGen），`intern()` 可能复制一份字符串到永久代；  
- JDK 7+：常量池移到堆，`intern()` 通常会把引用指向现有堆对象并记录到池中，内存行为不同。

【延伸考点】
- 大量对不同字符串调用 `intern()` 可能导致字符串池膨胀，需要结合业务评估。  
- 某些框架会对热点字符串（如枚举名、状态码等）做 `intern()` 以减少内存与比较开销。

### 变量


【问题】成员变量与局部变量的区别有哪些？

【参考答案】

1）语法和修饰符方面  
- **成员变量（字段）**：定义在类体中、方法外，可以被 `public` / `protected` / 默认 / `private`、`static`、`final`、`volatile`、`transient` 等修饰；  
- **局部变量**：定义在方法体、代码块或方法参数列表中，不能使用访问控制符和 `static`，但可以使用 `final`。

2）生命周期和存储位置（概念层面）  
- 成员变量：  
  - 实例字段的生命周期与对象一致；  
  - 静态字段的生命周期与类加载 / 卸载周期一致；  
  - 实际存储位置由 JVM 实现决定，通常作为对象的一部分存储在堆上。  
- 局部变量：  
  - 生命周期随方法调用/代码块执行而开始和结束；  
  - 通常分配在栈帧中，也可能因逃逸分析被优化为寄存器等。

3）默认值  
- 成员变量如果未显式赋值，会自动获得默认值（0、`false`、`null` 等；`final` 成员必须显式赋值）；  
- 局部变量**不会自动赋默认值**，在使用前必须显式初始化，否则编译不通过。

【延伸考点】
- 静态变量（类变量）与实例变量的区别与使用场景。  
- 局部变量表、操作数栈等 JVM 栈帧结构与局部变量的关系。



【问题】阐述静态变量和实例变量的区别？

【参考答案】

- **静态变量（类变量）**  
  - 使用 `static` 修饰，属于类本身；  
  - 在类加载时初始化，在类卸载前一直存在；  
  - 所有实例共享一份数据，可通过“类名.变量名”访问；  
  - 常用于全局配置、缓存、计数器等“类级别”状态（注意并发和内存泄漏问题）。

- **实例变量（成员变量）**  
  - 不使用 `static` 修饰，属于某个具体对象；  
  - 每创建一个实例，就有一份独立的实例变量副本；  
  - 需要通过对象引用访问，如 `obj.field`。

【延伸考点】
- 静态变量在并发场景下需要配合 `volatile`、锁等保证可见性与原子性。  
- 静态字段对 GC Root 的影响（可能导致对象无法被回收）。






### 引用


【问题】对象实体与对象引用有何不同？

【参考答案】

- **对象实例**：  
  - 通过 `new` 等方式在堆上分配的真实数据结构，包含对象头和实例字段，是“真正存数据的地方”。  

- **对象引用**：  
  - 保存“如何找到这个对象”的信息，可以理解为“指向对象的地址”或句柄；  
  - 引用变量本身可以位于不同地方：方法栈帧中的局部变量、堆中其他对象的字段、方法区中的静态变量等。

- 关系总结：  
  - 一个引用在同一时刻指向 0 个或 1 个对象（可以为 `null`）；  
  - 一个对象可以被多个引用指向，只要还有可达引用，GC 就不会回收该对象。

【延伸考点】
- JVM 中对象访问的两种主流实现：句柄池 vs 直接指针（HotSpot 采用后者）。  
- GC Roots 的定义：局部变量、静态变量、JNI 引用等。



【问题】强引用、软引用、弱引用、幻象引用有什么区别？具体使用场景是什么？

【参考答案】

不同引用类型体现了对象不同的“可达性”状态以及被 GC 回收的时机差异：

1）**强引用（StrongReference）**  
- 平时代码中最常见的引用方式，如：`Object obj = new Object();`；  
- 只要有强引用指向，对象就**不会被 GC 回收**；  
- 只有当所有强引用断开（如赋值为 `null`、超出作用域）且不可达时，对象才有机会被回收。

2）**软引用（SoftReference）**  
- 用 `SoftReference<T>` 包装对象；  
- 在内存足够时，软引用对象可以暂时保留；当 JVM 内存紧张、即将发生 OOM 时，会优先回收只被软引用关联的对象；  
- 典型用途：内存敏感缓存，如图片缓存。

3）**弱引用（WeakReference）**  
- 用 `WeakReference<T>` 包装对象；  
- 只要发生 GC，**弱引用关联的对象一定会被回收**（如果没有其他强/软引用）；  
- 典型用途：如 `WeakHashMap` 的 key、避免内存泄漏的监听器/回调列表等。  
- 在 Android/Java 中，为避免非静态内部类 Handler 导致的内存泄漏，也常用 `WeakReference<Activity>` 保存外部类引用。

4）**幻象引用 / 虚引用（PhantomReference）**  
- 不能通过 `get()` 取得对象，仅用于在对象“被 GC 确认要回收之后”得到一个通知；  
- 必须配合 `ReferenceQueue` 使用，用于实现更精细的资源释放或对象生命周期监控（Post-Mortem 清理）。

【延伸考点】
- 四种引用在 JDK `java.lang.ref` 包中的具体类与使用模式。  
- 软引用和弱引用在缓存中的实际取舍。  
- 为什么从 JDK 9 开始，某些与引用相关的实现做了性能优化（如 Cleaner 替代 finalizer）。




### 对象相等


【问题】对象的相等与指向它们的引用相等，两者有什么不同？

【参考答案】

- **引用相等（== 用于引用类型）**：  
  - 比较的是两个引用是否指向堆中的**同一个对象**，即“内存地址是否相同”；  
  - 对于字符串、包装类等引用类型，`a == b` 只表示是否是同一实例。

- **对象相等（`equals()`）**：  
  - 可以被重写，用来定义“业务意义上的相等”；  
  - 默认实现（`Object.equals`）也是比较引用相等；  
  - 常见类（如 `String`、集合、包装类等）重写了 `equals()`，用来比较内容相等。

【延伸考点】
- 在自定义实体类中重写 `equals()` / `hashCode()` 的规范与注意点。  
- 在集合（如 `HashSet`、`HashMap`）中，引用相等与对象相等分别起到什么作用。



【问题】== 与 equals 的区别？

【参考答案】

1）`==`  
- 对于**基本类型**：比较的是值是否相等；  
- 对于**引用类型**：比较的是两个引用是否指向同一对象（同一内存地址）。

2）`equals()`  
- `Object` 默认实现是 `==`，即比较引用是否相同；  
- 许多类（如 `String`、包装类、集合等）都重写了 `equals()`，用于比较“内容是否相等”；  
- 自定义类重写 `equals()` 时，通常需要同时重写 `hashCode()` 以保证集合行为正确。

【延伸考点】
- `equals()` 的自反性、对称性、传递性、一致性及与 `null` 的关系（Java 规范中的约定）。  
- 在集合中（尤其是 Hash 系列）`equals()` 与 `hashCode()` 协作的原理。



【问题】使用 equals 时如何避免空指针问题？

【参考答案】

- 避免这样写：`obj.equals("abc")`，当 `obj` 为 `null` 时会抛 `NullPointerException`；  
- 建议写成：`"abc".equals(obj)`，或使用 `Objects.equals(obj, "abc")`；  
- `Objects.equals(a, b)` 内部会做空判断，等价于：`(a == b) || (a != null && a.equals(b))`。

【延伸考点】
- 在重写 `equals()` 时，对待 `null` 的正确方式（永远返回 `false`，不要抛异常）。  
- 结合 `Comparator`、`Comparable` 使用时的空值处理策略。



【问题】16、两个对象值相同（x.equals(y) == true），但却可以有不同的 hashCode，这句话对不对？

【参考答案】

- **不对。**Java 规范要求：  
  1）如果 `x.equals(y) == true`，则必须有 `x.hashCode() == y.hashCode()`；  
  2）如果 `x.hashCode() == y.hashCode()`，并不一定有 `x.equals(y) == true`（允许哈希冲突）。  
- 因此，“相等对象必须有相等的哈希码”，而“相同哈希码的对象不一定相等”。

【延伸考点】
- 在重写 `equals()` 时，必须相应地重写 `hashCode()`，否则会导致集合行为异常（如相同元素存在多个副本）。  
- `hashCode()` 的实现应尽量让不同对象均匀分布，以减少哈希冲突。






### 基本数据类型


【问题】整形包装类型值如何比较？浮点类型数据如何比较？

【参考答案】

1）整型包装类型的比较（`Integer`、`Long` 等）
```java
Integer x = 3;  // 自动装箱
Integer y = 3;
System.out.println(x == y);        // true，落在缓存区间 [-128, 127]，引用相同

Integer a = 128;
Integer b = 128;
System.out.println(a == b);        // 通常为 false，超出缓存范围
System.out.println(a.equals(b));   // true，值相等
```java

- 原则：**不要依赖缓存，比较包装类型的数值相等应使用 `equals()` 或显式拆箱后用 `==`**，如 `x.intValue() == y.intValue()`。

2）浮点数的比较

- 由于二进制表示精度问题，直接用 `==` 比较浮点数往往不可靠：
```java
float a = 1.0f - 0.9f;
float b = 0.9f - 0.8f;
System.out.println(a);      // 0.100000024
System.out.println(b);      // 0.099999964
System.out.println(a == b); // false
```java

- 实战中常用两种方式：  
  - 使用**误差范围（epsilon）**比较：`Math.abs(a - b) < 1e-6`；  
  - 使用 `BigDecimal`（建议使用字符串构造），避免二进制浮点误差：

```java
BigDecimal a = new BigDecimal("1.0");
BigDecimal b = new BigDecimal("0.9");
BigDecimal c = new BigDecimal("0.8");
BigDecimal x = a.subtract(b); // 0.1
BigDecimal y = b.subtract(c); // 0.1
System.out.println(x.compareTo(y) == 0); // true，推荐使用 compareTo 比较数值
```

【延伸考点】
- `Integer.valueOf()` 缓存的范围可通过 JVM 参数调整（如 `-XX:AutoBoxCacheMax`）。  
- `BigDecimal.equals()` 与 `compareTo()` 的区别（前者会比较精度 Scale）。



【问题】a = a + b 与 a += b 有什么区别吗？

【参考答案】

- 对于表达式 `a = a + b`：  
  - 按 Java 语言规范，算术运算会进行类型提升，如 `byte`/`short` 在参与 `+` 运算时会先提升为 `int`；  
  - 表达式整体类型是提升后的类型（如 `int`），再赋值给左侧变量时如果类型不兼容，需要显式强转，否则编译错误。

- 对于表达式 `a += b`：  
  - 编译器会自动插入**隐式强制类型转换**，等价于：`a = (T)(a + b);`，其中 `T` 是 `a` 的类型；  
  - 因此在某些情况下 `a += b` 可以通过编译，而 `a = a + b` 不行：

```java
byte a = 127;
byte b = 1;
// b = a + b;      // 编译错误：int 不能隐式赋值给 byte
b += a;           // 等价于 b = (byte)(a + b); 编译通过
```

【延伸考点】
- 了解 Java 中的数值类型提升规则（binary numeric promotion）。  
- 注意避免利用 `+=` 的隐式强转掩盖溢出问题。



【问题】5.Java 支持的数据类型有哪些？什么是自动拆装箱？

【参考答案】

1）Java 的 8 种基本数据类型

```text
整数型：byte, short, int, long
浮点型：float, double
字符型：char
布尔型：boolean
```

对应的包装类型分别是：`Byte`、`Short`、`Integer`、`Long`、`Float`、`Double`、`Character`、`Boolean`。

2）自动装箱与自动拆箱

- **自动装箱（autoboxing）**：基本类型在需要对象类型的上下文中，编译器自动插入转换代码，如：
```java
Integer x = 10;        // 编译器生成：Integer x = Integer.valueOf(10);
List<Integer> list = new ArrayList<>();
list.add(5);           // 编译器生成：list.add(Integer.valueOf(5));

- **自动拆箱（unboxing）**：包装类型在需要基本类型的上下文中，编译器自动插入取值代码，如：

Integer x = 10;
int y = x;             // 编译器生成：int y = x.intValue();
```
【延伸考点】
- 自动拆箱时若包装类型为 `null` 会抛出 `NullPointerException`。  
- 装箱带来的性能问题（频繁创建对象）及整数缓存（`Integer.valueOf` 等）。  
- 在集合泛型中只能使用包装类型，不能使用基本类型。



【问题】4、float f = 3.4; 是否正确？

【参考答案】

- 直接写 `float f = 3.4;` **不正确**，`3.4` 是 `double` 类型字面量，将 `double` 赋给 `float` 属于窄化转换（down-casting），可能丢失精度，编译器要求显式强转；  
- 正确写法可以是：
```java
float f1 = 3.4F;       // 使用 float 字面量后缀 F 或 f
float f2 = (float) 3.4;
```
【延伸考点】
- 整数字面量、长整型（`L`/`l` 后缀）、浮点型（`F`/`f`、`D`/`d`）字面量规则。  
- 小写 `l`（L）容易与数字 1 混淆，建议使用大写 `L`。



【问题】11、switch 是否能作用在 byte 上，是否能作用在 long 上，是否能作用在 String 上？ 

【参考答案】

- 在 Java 5 以前：`switch (expr)` 中，`expr` 只能是 `byte`、`short`、`char`、`int`；  
- 从 Java 5 开始：引入枚举类型，`expr` 也可以是 `enum`；  
- 从 Java 7 开始：`expr` 还可以是字符串（`String`）；  
- 到目前为止，长整型（`long`）在所有版本中都**不可以**作为 `switch` 的表达式类型。

【延伸考点】
- `switch` 对应的字节码指令：`tableswitch`、`lookupswitch` 的差别。  
- 对字符串 `switch` 的编译实现（通常是编译成对 `hashCode()` 的 `switch` + `equals()` 校验）。



【问题】12、用最有效率的方法计算 2 乘以 8？ 

【参考答案】

- 可以使用位移运算：`2 << 3`（左移 3 位相当于乘以 \(2^3\)）；  
- 右移 3 位 `x >> 3` 则相当于对有符号整数除以 \(2^3\) 并向下取整。

【补充说明】
- 在现代 JVM / JIT 优化下，简单的乘法 / 除法通常会自动转成高效指令，**在业务代码中直接写 `2 * 8` 即可，可读性更好**；  
- 面试更多考察的是你对二进制和位运算原理的理解。



【问题】Java 的八种基本数据类型，每个占多少个字节？

【参考答案】

- 位（bit）：计算机中存储数据的最小单位，只能表示 0 或 1；  
- 字节（byte）：由 8 个 bit 组成，是内存容量的基本单位；`1 Byte = 8 bit`。

Java 8 种基本类型的位数和字节数（由语言规范固定，不随平台变化）：

```text
byte    8  bit = 1 byte
short   16 bit = 2 byte
int     32 bit = 4 byte
long    64 bit = 8 byte
float   32 bit = 4 byte
double  64 bit = 8 byte
char    16 bit = 2 byte
boolean 理论上仅 1 bit，但 JVM 具体实现可能按 1 byte 或更大对齐存储
```

【延伸考点】
- `boolean` 在数组或对象中的实际布局是由 JVM 实现决定的，规范只规定语义。  
- 不同语言中基本类型大小不一致的情况（如 C/C++）与 Java 的对比。



【问题】short s1 = 1; s1 = s1 + 1; 有什么错？`short s1 = 1; s1 += 1;` 是否有错？

【参考答案】

1）`short s1 = 1; s1 = s1 + 1;`  
- `short s1 = 1;` 没有错误，`1` 是 `int` 字面量，但编译器会检测常量范围，允许窄化赋值给 `short`；  
- 表达式 `s1 + 1` 会进行数值提升，结果类型是 `int`，再赋给 `short` 时需要显式强转，否则编译错误。

正确写法：

```java
short s1 = 1;
s1 = (short) (s1 + 1);
```

2）`short s1 = 1; s1 += 1;`  
- 这段代码**没有错误**，`+=` 会自动做隐式强制类型转换，等价于：

```java
short s1 = 1;
s1 = (short) (s1 + 1);
```

【延伸考点】
- 与前面 `a = a + b` 与 `a += b` 的问题是同一类考点。  
- 编译期常量表达式与数值范围检查。



【问题】Math.round(11.5) 等于多少？Math.round(-11.5) 等于多少？

【参考答案】

```java
Math.round(11.5);   // 12
Math.round(-11.5);  // -11
```

解释：`Math.round(float a)` / `Math.round(double a)` 本质上是对参数加上 0.5 后向下取整，但**对负数的舍入方向要特别注意**：

- `Math.round(x)` 等价于 `(long) Math.floor(x + 0.5d)`；  
- `Math.round(11.5)` → `floor(12.0)` → 12；  
- `Math.round(-11.5)` → `floor(-11.0)` → -11（不是 -12）。

【延伸考点】
- `Math.floor`、`Math.ceil`、`Math.rint`、`BigDecimal` 的舍入模式差异（如 HALF_UP、HALF_EVEN）。  
- 金融场景下推荐使用 `BigDecimal` 配合明确的舍入模式。



【问题】Java 当中使用什么类型表示价格比较好?

【参考答案】

- 在**金额、价格**等对精度要求高的场景，推荐使用 `BigDecimal` 表示，避免二进制浮点的精度误差，并结合适当的舍入模式；  
- 只有在对性能 / 内存极端敏感、且可容忍少量精度损失的场景，才考虑使用 `double` 并配合业务逻辑控制误差。

【延伸考点】
- `BigDecimal` 的构造方式：推荐使用 `String` 或 `valueOf(double)`，避免直接使用 `new BigDecimal(double)` 带来的隐性误差。  
- 电商、金融系统中金额字段的常见表示方式（如“以分为单位的 long” vs `BigDecimal`）。



【问题】可以将 int 强转为 byte 类型么？会产生什么问题？

【参考答案】

- 可以进行强制类型转换：`byte b = (byte) someInt;`；  
- 但 Java 中 `int` 是 32 位、`byte` 是 8 位，强制转换时高 24 位会被截断，只保留低 8 位，因此可能发生**溢出与符号变化**。

示例：

```java
int i = 130;
byte b = (byte) i;  // 130 的二进制为 1000_0010，截断后对应的 byte 值为 -126
System.out.println(b); // 输出 -126
```

【延伸考点】
- Java 整型采用二进制补码表示，负数的高位为 1。  
- 各类强制类型转换的规则与可能产生的数据截断问题。



【问题】数据类型之间的转换？

【参考答案】

1）字符串转基本数据类型 / 包装类型

- 使用对应包装类的 `parseXxx(String)` 或 `valueOf(String)` 方法，例如：

```java
int i = Integer.parseInt("123");
long l = Long.parseLong("456");
double d = Double.parseDouble("1.23");
boolean b = Boolean.parseBoolean("true");
Integer obj = Integer.valueOf("123");
```

2）基本数据类型转字符串

- 使用 `String.valueOf(...)`：

```java
String s1 = String.valueOf(123);
String s2 = String.valueOf(1.23);
```

- 或与空字符串拼接（不推荐在性能敏感代码中大量使用）：

```java
String s3 = 123 + "";
```

【延伸考点】
- 解析失败时的异常：如 `Integer.parseInt("abc")` 抛出 `NumberFormatException`；  
- `Boolean.parseBoolean("true")` 仅在字符串忽略大小写等于 `"true"` 时返回 `true`。






### 序列化


【问题】Java 序列化中如果有些字段不想进行序列化，怎么办？

【参考答案】

- 对不希望被序列化的实例字段，使用 `transient` 关键字修饰：  
  - 序列化时，这些字段的值不会被写入流中；  
  - 反序列化时，这些字段会被赋予类型默认值（或由自定义反序列化逻辑重新初始化）。  
- `transient` 只能修饰**变量**（实例字段或静态字段），不能修饰类和方法。

【延伸考点】
- 配合 `static` 使用：`static` 字段本身不会被序列化，但可以同时标记为 `transient` 表意更清晰。  
- 对于敏感信息（如密码、密钥），既可以设置为 `transient`，也可以使用自定义序列化逻辑加密处理。



【问题】什么是 Java 序列化，如何实现 Java 序列化？

【参考答案】

1）什么是序列化 / 反序列化
- **序列化（Serialization）**：将对象状态转换为字节序列，以便存储到磁盘、在网络中传输等；  
- **反序列化（Deserialization）**：从字节序列中恢复出 Java 对象。

2）如何实现 Java 序列化
- 让类实现标记接口 `java.io.Serializable`，例如：

```java
public class User implements Serializable {
    private static final long serialVersionUID = 1L;
    private String name;
}
```

- 建议**显式声明 `serialVersionUID`**，避免类结构变更导致反序列化失败（`InvalidClassException`）。

3）常见序列化方式
- JDK 原生序列化：`ObjectOutputStream` / `ObjectInputStream`；  
- 自定义序列化逻辑：实现 `writeObject` / `readObject` / `readResolve` 等方法；  
- 实际项目中更常用 JSON、Protobuf、Kryo 等替代方案，性能和可读性更好。

【延伸考点】
- JDK 序列化的安全问题（反序列化漏洞）与黑名单机制。  
- 为什么 JDK 官方多次建议避免在新系统中继续依赖原生序列化。






### 静态



【问题】3. “static” 关键字是什么意思？Java 中是否可以覆盖（override）一个 static 方法？

【参考答案】

1）`static` 的含义
- 修饰字段：表示**类变量**，属于类本身而非某个实例；  
- 修饰方法：表示**类方法**，可以通过“类名.方法名”直接调用，不依赖具体实例；  
- 修饰代码块：静态代码块，在类加载时执行一次；  
- 修饰内部类：静态内部类，不依赖外部类实例。

2）static 方法能否被覆盖？
- 严格来说，**static 方法不能被“重写（override）”，只能被“隐藏（hide）”**：  
  - 方法覆盖是基于运行时动态绑定的实例方法机制；  
  - static 方法在编译期就已静态绑定，与具体实例无关。  
- 如果子类中声明了与父类同名同参数列表的 static 方法，实际上是**隐藏**了父类的该方法，调用时根据“引用的编译时类型”决定调用哪个。

【延伸考点】
- 静态方法中不能直接访问实例成员（因为没有 `this`），必须通过对象引用访问。  
- 尽量避免在多态层次结构中“重名的 static 方法”，否则容易引发理解混乱。



【问题】4. 是否可以在 static 环境中访问非 static 变量？

【参考答案】

- 在纯静态环境（如 static 方法、静态代码块）中，**不能直接访问非 static 成员**，因为此时可能尚未创建任何实例，也不存在 `this` 引用；  
- 但可以通过**已创建的对象引用**来访问实例成员，例如：

```java
static void foo() {
    MyClass obj = new MyClass();
    System.out.println(obj.instanceField);
}
```

【延伸考点】
- `main` 方法是静态方法，因此在 `main` 中直接访问实例成员会导致编译错误，必须先创建对象。  
- 设计上尽量减少滥用 `static`，避免可测试性与扩展性变差。



【问题】抽象方法是否可以同时是 static？是否可同时是本地方法（native）？是否可同时被 synchronized 修饰？

【参考答案】

- 都不可以。原因如下：  
  - 抽象方法（`abstract`）表示“只有声明，没有实现，需要子类重写”；  
  - `static` 方法是与类相关的、在编译期静态绑定的方法，不参与子类的动态绑定覆盖机制，与“多态重写”概念冲突；  
  - `native` 方法由本地代码（如 C/C++）实现，Java 端不提供方法体，而抽象方法是“没有实现”的声明，二者语义重复冲突；  
  - `synchronized` 用于修饰有具体实现的方法或代码块，以便在运行时加锁，抽象方法没有实现细节，因此不能加锁。

【延伸考点】
- 组合使用修饰符时要遵守语言规范，如 `abstract` 与 `final` 也不能同时用于方法。  
- 接口中方法在不同 Java 版本下隐式修饰符的演化（Java 8 的 default / static 方法等）。






### 方法与函数


【问题】6. Java 中的方法覆盖（Overriding）和方法重载（Overloading）是什么意思？

【参考答案】

1）方法重载（Overloading）
- 发生在**同一个类中**（或父子类链上），方法名相同，但**参数列表不同**（参数个数或类型不同）；  
- 与返回值类型无关（仅修改返回值不能构成重载）；  
- 是**编译期多态**（静态绑定），编译器根据实参列表选择合适的方法签名。

2）方法覆盖（Overriding）
- 发生在**子类中重新定义父类方法**的场景；  
- 要求：方法名、参数列表完全相同，返回类型相同或是协变返回类型（子类型）；  
- 访问修饰符不能更严格（可以放宽，如 `protected` → `public`）；  
- 只能针对实例方法，不能重写 `static` 方法；  
- 是**运行时多态**（动态绑定），根据实际对象类型决定调用哪个实现。

【延伸考点】
- `@Override` 注解的作用（帮助编译器校验签名是否真正覆盖）。  
- 构造方法不能被重写，只能被调用（`super(...)`）。



【问题】7. Java 中，什么是构造函数？什么是构造函数重载？什么是复制构造函数？

【参考答案】

1）构造函数
- 与类名相同、没有返回类型（连 `void` 也不能有）的方法；  
- 在创建新对象时由 `new` 调用，用于完成对象的初始化工作；  
- 如果程序员没有显式定义任何构造函数，编译器会自动生成一个**无参、无任何逻辑**的默认构造函数。

2）构造函数重载
- 与方法重载类似，一个类中可以定义多个构造函数，只要**参数列表不同**；  
- 可以通过不同的构造方法提供不同的初始化方式，内部常见写法是使用 `this(...)` 进行构造器链调用以复用逻辑。

3）复制构造函数
- Java 不像 C++ 那样有语言级别的“默认复制构造函数”；  
- 但我们可以**手动编写一个以本类对象为参数的构造函数**来实现“拷贝构造”的效果，例如：

```java
public class User {
    String name;
    public User(User other) {
        this.name = other.name;
    }
}
```

【延伸考点】
- 无参构造器在框架（如 Spring、JPA、序列化）中的作用。  
- 构造器 vs 静态工厂方法（Effective Java 推荐的创建对象方式）。



【问题】13、数组有没有 length() 方法？String 有没有 length() 方法？ 

【参考答案】

- 数组：通过**属性** `arr.length` 获取长度（不是方法，没有括号）；  
- `String`：通过**方法** `str.length()` 获取长度；  
- 集合：通过 `size()` 方法获取元素个数，如 `list.size()`、`set.size()`；  
- 文件：`File.length()` 返回文件大小（字节数）。



【问题】14、在 Java 中，如何跳出当前的多重嵌套循环？ 

【参考答案】

- 可以使用**带标签（label）的 break** 语句，在最外层循环前加一个标签，然后 `break 标签名;`：

```java
public class TestBreak {
    public static void main(String[] args) {
        outfor: for (int i = 0; i < 10; i++) {
            for (int j = 0; j < 10; j++) {
                if (j == 5) {
                    break outfor; // 直接跳出外层循环
                }
                System.out.println("j = " + j);
            }
        }
    }
}
```

【补充说明】
- Java 支持带标签的 `break` 和 `continue`，在语义上有点类似 C/C++ 的 `goto`；  
- 实际编码中应谨慎使用标签，避免影响代码可读性，能通过重构提取方法等方式替代时尽量不用。



【问题】Java 在静态方法中可以调用哪些方法？

【参考答案】

- 不能使用 `this` 或 `super`，因为静态方法不依赖具体实例；  
- 可以直接调用本类的其他静态方法，也可以通过“类名.方法名”调用其他类的静态方法；  
- 如需调用实例方法，必须先创建对象，再通过对象引用调用，例如：

```java
static void foo() {
    MyClass obj = new MyClass();
    obj.bar(); // bar 是实例方法
}
```

【延伸考点】
- 静态工具类（如 `Collections`、`Math`）的设计模式与适用场景。  
- 静态方法在测试和扩展性上的局限性。






### 接口与抽象类


【问题】8. Java 支持多继承么？Java 8 之后支持吗？

【参考答案】

1）类的继承
- Java **不支持类的多继承**，每个类只能有一个直接父类；  
- 如果同时出现继承和实现，语法顺序必须是：`class A extends B implements C, D {}`。

2）接口的多实现与默认方法
- 一个类可以实现多个接口，实现了“类型层面”的多继承；  
- 自 Java 8 起，接口允许定义带实现的 `default` 方法和 `static` 方法：  
  - 当一个类实现多个接口，而这些接口中包含**同名同参数列表的 default 方法**时，编译器要求实现类显式覆盖该方法以解决冲突；  
  - 也可以在实现类中通过 `接口名.super.methodName()` 指定调用某个接口的默认实现。

3）总结
- Java 类本身仍然遵循**单继承**原则；  
- 通过接口多实现 + default 方法，Java 在行为层面提供了有限的“多继承”能力。

【延伸考点】
- 多继承 vs 接口多实现的优劣，及其与菱形继承问题的关系。  
- 在接口中合理使用 default 方法的场景与反例。



【问题】9. 接口和抽象类的区别是什么？

【参考答案】

1）设计意图
- 接口：更侧重**定义能力 / 契约**，强调“能做什么”；  
- 抽象类：更侧重**代码复用与部分实现**，同时提供“是什么 + 怎么做”的一部分。

2）方法与实现
- 接口（Java 8+）：  
  - 可以有抽象方法、`default` 方法、`static` 方法；  
  - 所有抽象方法默认为 `public abstract`。  
- 抽象类：  
  - 可以包含抽象方法和非抽象方法；  
  - 可以有构造函数、普通字段、静态代码块等。

3）字段
- 接口中的字段默认是 `public static final`，即**常量**，必须赋初始值；  
- 抽象类中可以有各种访问级别的实例字段和静态字段。

4）继承关系
- 一个类只能继承一个抽象类，但可以实现多个接口；  
- 抽象类可以实现接口，也可以在未实现全部接口方法时保持抽象。

5）实例化
- 接口和抽象类都不能直接实例化，但抽象类可以拥有 `main` 方法、静态方法并被调用。

【延伸考点】
- 选择接口还是抽象类的判断标准：是否需要共享状态 / 部分实现、是否需要多实现等。  
- “接口隔离原则”“依赖倒置原则”与接口 / 抽象类的关系。



【问题】接口可以继承接口？抽象类能实现接口？抽象类能继承实体类？

【参考答案】

- 接口可以继承接口：  
  - 如 `List` 继承 `Collection`，一个接口可以继承多个接口。

- 抽象类可以实现接口：  
  - 常见于“适配器模式”，抽象类为接口提供部分默认实现，子类按需选择性重写。

- 抽象类可以继承实体类：  
  - 所有抽象类最终都继承 `Object`，也可以继承其他非抽象类，前提是语义合理。





【问题】Java 8 之后对接口做了什么改动？

【参考答案】

1）默认方法（Default Methods）
- 概念：允许在接口中定义带有**默认实现**的方法，实现类可以选择使用或覆盖；  
- 语法：使用 `default` 关键字，例如：

public interface MyInterface {
    default void myMethod() {
        System.out.println("Default implementation");
    }
}

- 作用：支持接口“向后兼容演进”。为已有接口新增方法时，可以给出默认实现，避免破坏所有现有实现类的编译。
- 冲突解决：如果一个类实现的多个接口中包含**同名同参的默认方法**，编译器会报错，要求在实现类中显式覆盖并选择具体实现，或用 `接口名.super.methodName()` 指定调用哪个接口的默认方法。

2）静态方法（Static Methods）
- 概念：接口可以声明 `static` 方法，这些方法属于接口本身，而不是实例；  
- 语法示例：

public interface MyInterface {
    static void myStaticMethod() {
        System.out.println("Static method in interface");
    }
}

- 调用方式：只能通过 `接口名.静态方法名()` 调用，不能通过实现类实例调用。
- 作用：为接口提供与之紧密相关的**工具方法 / 工厂方法**，避免把这些方法塞进不相关的工具类里。

3）意义与影响
- 提高接口的**表达能力和可维护性**，支持 API 平滑演进；  
- 配合 Lambda、函数式接口（`@FunctionalInterface`），增强对函数式编程风格的支持；  
- 简化部分设计模式（如策略、模板方法）的实现，不再必须依赖抽象类做默认实现。

【延伸考点】
- Java 9 接口中新引入的 **私有方法（private method）**，用于在接口内部复用逻辑。  
- default 方法滥用带来的可读性 / 设计问题，以及与抽象类职责的边界。






### 值传递

【问题】10. 什么是值传递和引用传递？Java 是哪一种？

【参考答案】

1）概念区分  
- **值传递（pass by value）**：方法拿到的是实参值的一个拷贝，对形参的修改不会影响实参本身；  
- **引用传递（pass by reference）**：方法拿到的是实参本身的别名，对形参的修改会直接作用在实参上（包括改变它指向的对象）。

2）Java 的参数传递机制  
- Java **只有值传递**；  
- 对于基本类型，传递的是“数值”的拷贝；  
- 对于引用类型，传递的是“引用（指针）”的拷贝：  
  - 可以通过这个引用拷贝修改它指向对象的内部状态；  
  - 但如果在方法内让引用指向新的对象，不会影响方法外原来的引用。

可以总结为：  
- 修改基本类型形参的值：不会影响外部变量；  
- 修改对象形参引用指向的新对象：不会改变外部引用的指向；  
- 修改对象 / 集合内部的字段或元素：会影响外部看到的同一个对象 / 集合。

【延伸考点】
- 结合实际代码理解：方法内对 `list.add(...)` 与 `list = new ArrayList<>()` 的差异。  
- 与 C/C++ 中指针 / 引用语义的对比。

【问题】18. 当一个对象被当作参数传递到一个方法后，此方法可改变这个对象的属性，并可返回变化后的结果，那么这里到底是值传递还是引用传递？ 

【参考答案】

- 这是**值传递**，但传递的是“引用的值（地址）”；  
- 形参和实参都指向堆上的同一个对象，因此在方法中通过形参修改对象属性，外部也能观察到变化；  
- 但如果在方法内部给形参重新赋值（让它指向另一个新对象），方法外原始引用并不会改变。

一句话概括：**Java 只有值传递，引用类型参数也是“引用值”的拷贝**。

【延伸考点】
- 典型面试代码题：交换两个对象引用为什么在 Java 中实现不了。  
- 如何设计 API，避免因可变参数对象引起的副作用。



### 类加载

【问题】21. 描述一下 JVM 加载 class 文件的原理机制？ 

【参考答案】

1）类加载的触发与职责  
- JVM 中类的装载由 **类加载器（ClassLoader）** 及其子类完成，负责在运行时查找并装入 `.class` 字节码；  
- Java 采用**按需加载**机制：只有在首次主动使用某个类时才触发加载，而不是一次性加载所有类，从而节省内存、加快启动。

2）常见的加载方式  
- 隐式加载：在代码中首次使用某个类（如 `new`、访问静态字段、调用静态方法）时，由 JVM 自动触发加载；  
- 显式加载：通过 `Class.forName()`、`ClassLoader.loadClass()` 等反射方式手动触发加载。

【延伸考点】
- 哪些动作会触发类的“主动使用”（如 new、访问静态常量、反射等），从而引发初始化。  
- 与类加载相关的异常：`ClassNotFoundException`、`NoClassDefFoundError` 的区别。

【问题】类加载过程是怎样的？

【参考答案】

Java 虚拟机规范将类加载过程分为三个大阶段：**加载（Loading）→ 连接（Linking）→ 初始化（Initialization）**。

1）加载（Loading）  
- 通过类全限定名获取字节流（来自文件、网络或自定义源）；  
- 将字节流表示的静态存储结构转换为方法区 / 元空间中的运行时数据结构；  
- 在内存中生成代表该类的 `Class` 对象，作为程序访问类元数据的入口。

2）连接（Linking）：包含三个子阶段  
- 验证（Verification）：确保字节码格式、语义正确、安全（比如栈深度、类型安全等）；  
- 准备（Preparation）：为静态变量分配内存并设置默认初始值（`final static` 常量可能在编译期就放入常量池）；  
- 解析（Resolution）：将常量池中的符号引用替换为直接引用（如方法、字段的实际内存地址），部分 JVM 会延迟解析。

3）初始化（Initialization）  
- 执行类变量的显式赋值和静态代码块（`<clinit>`），按源码顺序执行；  
- 该阶段是真正执行类初始化逻辑的地方，且只会**执行一次**。

【延伸考点】
- 初始化阶段的线程安全保证（同一个类只会被一个线程初始化，其他线程阻塞等待）。  
- 类加载过程与双亲委派模型、类卸载之间的关系。

【问题】类加载器有哪些？什么是双亲委派模型？

【参考答案】

1）常见类加载器  
- BootstrapClassLoader（启动类加载器）：由 C/C++ 实现，负责加载 `JAVA_HOME/lib` 目录下的核心类库或 `-Xbootclasspath` 指定的类；  
- ExtensionClassLoader（扩展类加载器）：加载 `JAVA_HOME/lib/ext` 或 `java.ext.dirs` 指定路径下的扩展类库；  
- AppClassLoader（应用程序类加载器）：加载应用 `classpath` 下的所有类和第三方库，是日常最常用的类加载器；  
- 自定义 ClassLoader：应用可以通过继承 `ClassLoader` 定制自己的加载逻辑（如从网络、加密包加载）。

2）双亲委派模型（Parent Delegation Model）  
- 当某个类加载器收到类加载请求时，**优先把请求委派给父加载器处理**：  
  - 若父加载器可以完成加载，则直接返回；  
  - 只有在父加载器无法加载时，子加载器才尝试自己加载。  
- 这样可以确保：  
  - 核心类库由顶层加载器统一加载，避免用户代码“伪造”重要类（如 `java.lang.String`）；  
  - 同一个类不会被不同类加载器重复加载，保证类型一致性。

【延伸考点】
- 打破双亲委派模型的典型场景（如 SPI、应用服务器 / 插件系统中的热部署）。  
- 不同类加载器加载的同名类互相不兼容的现象（类隔离）。



### 对象

【问题】String s = new String("xyz"); 创建了几个 String 对象？

【参考答案】

- 字面量 `"xyz"` 在类加载或首次使用时会在**字符串常量池**中创建一个 `String` 对象；  
- 表达式 `new String("xyz")` 一定会在堆上再创建一个新的 `String` 对象，其内容拷贝自字面量对象；  
- 因此，这行代码总共创建了 **2 个** `String` 对象（常量池 1 个，堆上 1 个）。

【延伸考点】
- `new String("xyz")` 和 `"xyz"` 在内存位置与比较（`==` / `equals`）上的差异。  
- `String.intern()` 对上述场景中对象数量和引用关系的影响。

【问题】怎样创建一个 immutable（不可变）类？

【参考答案】

常见做法包括：
- 将类声明为 `final`，防止被继承后通过子类破坏不可变性；  
- 将所有成员变量声明为 `private` 且尽量 `final`，不提供修改状态的 setter 方法；  
- 构造函数中对可变对象参数进行**防御性拷贝**（deep copy），而不是直接持有外部引用；  
- 对外提供的 getter，如返回集合或数组时，返回只读视图或副本，而不是内部可变对象本身；  
- 如果类中确实需要可变状态，内部通过复制新对象的方式返回修改后的实例（类似于 `String` 的做法）。

【延伸考点】
- JDK 中的典型不可变类：`String`、装箱类型（`Integer` 等）、`BigDecimal` 等。  
- 不可变对象在并发编程中的优势（线程安全、易于缓存、无锁共享）。

【问题】String s = "Hello"; s = s + " world!"; 这两行代码执行后，原始的 String 对象中的内容变了没有？

【参考答案】

- 没有。`String` 是不可变类，**一旦创建其内容就不能改变**；  
- 第一行：`s` 指向常量池或堆中的一个 `String` 对象，内容为 `"Hello"`；  
- 第二行：`s + " world!"` 会创建一个新的 `String` 对象，内容为 `"Hello world!"`，然后让变量 `s` 指向新对象；  
- 原来内容为 `"Hello"` 的 `String` 对象仍然存在于内存中，只是 `s` 不再指向它。

【延伸考点】
- 大量使用 `+` 拼接字符串时产生多余中间对象的问题，以及 `StringBuilder` 的优化作用。  
- 对不可变对象变量重新赋值与对象本身被修改的区别。

【问题】下面这段关于字符串常量池和 `==` 比较的代码，分别输出什么？为什么？

String str1 = "hello";           // 字符串常量池中
String str2 = "he" + new String("llo"); // 运行期拼接，结果在堆中
String str3 = "he" + "llo";      // 编译期常量折叠，常量池中

System.err.println(str1 == str2);
System.err.println(str1 == str3);

String s1 = "ab", s2 = "a" + "b", s3 = "a", s4 = "b", s5 = s3 + s4;

【参考答案】

- `str1 == str2` 为 `false`：`str1` 指向常量池中的 `"hello"`，`str2` 是运行时拼接产生的新对象，位于堆中；  
- `str1 == str3` 为 `true`：`"he" + "llo"` 在编译期被折叠为 `"hello"`，与 `str1` 共享同一个常量池对象；  
- 对于 `s1` / `s2` / `s5`：  
  - `s1` 和 `s2` 都是编译期常量 `"ab"`，指向常量池中的同一个对象；  
  - `s5 = s3 + s4` 是运行期拼接，结果是堆中新建的 `String` 对象；  
  - 因此 `s5 == s2` 为 `false`。

【延伸考点】
- 编译期常量折叠（constant folding）与运行期字符串拼接的区别。  
- 使用 `equals()` 比较字符串内容 vs 使用 `==` 比较引用的适用场景。





### 运算

【问题】介绍一下 Java 中位运算 ^、&、<<、>>、>>>。

【参考答案】

- `^`（按位异或）：二进制位相同为 0，不同为 1；  
- `&`（按位与）：两个二进制位都为 1 时结果为 1，否则为 0；  
- `|`（按位或）：两个二进制位有一个为 1 即为 1；  
- `<<`（左移）：将二进制位整体左移 n 位，右侧空位补 0，相当于乘以 \(2^n\)（可能溢出）；  
- `>>`（算术右移）：保留符号位（最高位），右移时左侧空位补符号位，对有符号整数相当于除以 \(2^n\) 并向下取整；  
- `>>>`（无符号右移）：忽略符号位，左侧空位一律补 0，对负数右移后会变成正数。

【延伸考点】
- 位运算在权限标志（bit mask）、哈希运算、快速乘除中的应用。  
- 理解 `hash & (length - 1)` 与 `% length` 的关系（典型于 HashMap 中）。

【问题】"|" 与 "||" 的区别？

【参考答案】

- `|`：按位或运算符；用于布尔表达式时是**非短路或**，左右操作数都会计算；  
- `||`：逻辑或运算符，具有**短路行为**：  
  - 先计算左侧表达式；  
  - 如果左侧已能决定结果为 `true`，右侧不会再计算；  
  - 只有左侧为 `false` 时才计算右侧。

【延伸考点】
- 对于条件中包含函数调用或副作用代码时，`|` 与 `||` 的行为差异。  
- 同理还有 `&` vs `&&` 的区别。

【问题】int 型除以 double 型，结果是什么类型？有哪些自动类型转换规则？

【参考答案】

- 在表达式中，`int` 与 `double` 运算时，`int` 会被提升为 `double`，因此**结果类型是 `double`**；  
- 一般规则：  
  1）若参与运算的类型不同，先转换成同一类型再进行运算；  
  2）转换按“精度不丢失”的方向进行，优先向位数更高、表示范围更大的类型转换；  
  3）所有浮点运算都会提升为双精度 `double` 进行（即使两个都是 `float`）。

【延伸考点】
- `byte` / `short` / `char` 在算术运算中会先提升为 `int` 的规则。  
- 赋值运算时的隐式窄化不会发生，需显式强转（与前面 `a+=b` 整体考点对应）。

【问题】三目运算符的空指针问题是怎么回事？

【参考答案】

示例：

boolean flag = true;
boolean simpleBoolean = false;
Boolean nullBoolean = null;
boolean x = flag ? nullBoolean : simpleBoolean;

上述代码会抛出 `NullPointerException`，原因是：
- 三目运算符会根据第二、第三个操作数的类型推断结果类型：  
  - 当二、三位操作数类型分别为包装类型和基本类型时，结果类型倾向于**基本类型**；  
- 编译器会对 `nullBoolean` 做自动拆箱，转成 `nullBoolean.booleanValue()`，此时因 `nullBoolean` 为 `null` 导致 NPE。

Java 8 之前类似表达式：

Map<String, Boolean> map = new HashMap<>();
Boolean b = (map != null ? map.get("hello") : false);

会因拆箱规则等导致 NPE；而在 Java 8 及以后，类型推断更严格，上例的结果会是 `null`（视实现而定）。

【延伸考点】
- 编写三目表达式时尽量确保第二、三操作数类型一致，并对可能为 `null` 的包装类型先做判空。  
- 使用 `Objects.requireNonNull` 或 `Boolean.TRUE.equals(...)` 等方式规避 NPE。





### 拷贝

【问题】深拷贝和浅拷贝的区别是什么？

【参考答案】

- **浅拷贝（shallow copy）**：  
  - 被复制对象的字段值会被复制；  
  - 对于引用类型字段，只复制“引用”（指针），新旧对象内部字段仍指向同一个子对象；  
  - 修改子对象状态，会同时影响原对象与拷贝对象。

- **深拷贝（deep copy）**：  
  - 不仅复制当前对象的字段，还会递归复制其引用指向的子对象；  
  - 生成独立对象图，新旧对象互不影响；  
  - 成本更高（时间/内存）。

【延伸考点】
- 深拷贝常见实现：拷贝构造器/工厂方法、手写递归复制、序列化-反序列化、第三方库等。  
- 集合（例如 `List<List<...>>`）只做浅拷贝很容易踩坑。

【问题】解析 XML 文档有哪几种方式？

【参考答案】

- **DOM**：一次性加载整个 XML 到内存，构建树结构；便于随机访问/修改，但内存占用高。  
- **SAX**：事件驱动的流式解析，从头到尾顺序读取；内存占用低，适合大文件，但需要自己维护解析状态。

【延伸考点】
- **StAX**：拉模式（pull）解析，与 SAX（push）相对。  
- 现代项目常用 JSON，但 XML 在老系统与配置（如 Maven、Spring 早期配置）里仍常见。



### 关键字

【问题】final 关键字除了修饰类之外，还有哪些用法？

【参考答案】

- 修饰类：类不可被继承（如 `String`）。  
- 修饰方法：方法不可被子类覆盖（override），但可以被重载（overload）。  
- 修饰变量（字段 / 局部变量 / 参数）：  
  - 只能赋值一次；  
  - 实例字段必须在声明处、构造器或初始化块中完成初始化；  
  - 常与 `static` 一起定义常量：`public static final int MAX = 10;`。

【延伸考点】
- `final` 不能与 `abstract` 联用（语义冲突）。  
- 并发语义：`final` 字段在构造完成后的安全发布特性（了解 JMM）。

【问题】final 是不是 immutable？

【参考答案】

- **final 不等同于 immutable**：  
  - `final` 限制的是“引用不能再指向其他对象”；  
  - 不限制“对象内部状态是否可变”。对象如果是可变类，仍然能修改其内部内容。

示例：

```java
final List<String> strList = new ArrayList<>();
strList.add("Hello");
strList.add("world");

List<String> unmodifiableStrList = List.of("hello", "world");
unmodifiableStrList.add("again"); // 运行时抛异常
```

【延伸考点】
- 不可变对象的设计要点：final 类、私有 final 字段、防御性拷贝、只读访问等。  
- `Collections.unmodifiableList` vs `List.of` 的差异（只读视图 vs 不可变集合）。

## java线程

【问题】11. 进程和线程的区别是什么？

【参考答案】

- **进程**：操作系统进行资源分配与调度的基本单位，拥有独立的地址空间和系统资源。  
- **线程**：进程内的执行单元，共享进程内存与资源，创建与切换成本通常更低。  
- 一个进程可包含多个线程并发执行；线程常被称为“轻量级进程”（面试常用说法）。

【延伸考点】
- JVM 是一个进程，Java 线程属于该进程。  
- 多进程隔离强但通信更复杂；多线程共享资源但更容易出现并发问题。

【问题】并发优势与并发风险分别有哪些？

【参考答案】

- **优势**：  
  - 提升吞吐/响应：并行处理多个任务；  
  - 提高资源利用率：I/O 等待期间 CPU 可执行其他任务；  
  - 更贴合某些业务模型：如生产者-消费者、异步处理等。
- **风险**：  
  - **安全性**：数据竞争导致结果不符合预期；  
  - **活跃性**：死锁、活锁、饥饿等导致无法推进；  
  - **性能**：线程过多引发频繁上下文切换、锁竞争、内存占用增加。

【延伸考点】
- 并发三大问题：安全性、活跃性、性能。  
- 线程池用于控制线程数量、降低创建开销。

【问题】12. 创建线程有几种方式？更推荐哪一种？为什么？

【参考答案】

- 从“任务执行单元”的角度：  
  - `Runnable`：无返回值；  
  - `Callable`：有返回值，可抛受检异常（配合 `Future`/线程池）。  
- 从“线程创建”的角度：最终仍由 `Thread` 或线程池创建线程执行任务。

更推荐：**`Runnable/Callable` + 线程池（ExecutorService）**  
- 避免继承 `Thread`（Java 单继承限制）；  
- 线程生命周期与资源可控，易于复用，提高吞吐并降低开销。

【延伸考点】
- `submit(Callable)` 返回 `Future`，可获取结果与异常。  
- 区分“线程（Thread）”与“任务（Runnable/Callable）”。

【问题】如何正确地停止线程？

【参考答案】

- 使用协作式中断：调用 `interrupt()` 发出中断信号；  
- 在线程内部定期检查 `isInterrupted()` 或捕获 `InterruptedException`，在合适位置退出并清理资源；  
- 不要使用 `stop`、`suspend`、`resume`（已废弃，可能导致资源不一致、死锁等）。

【延伸考点】
- `interrupt()` 不是强制终止，任务代码必须“响应中断”才会停止。  
- 线程池停止常用 `shutdown()` / `shutdownNow()`（理解差异）。

【问题】实现 Runnable 接口和 Callable 接口的区别？

【参考答案】

- `Runnable#run()`：无返回值，不能抛受检异常；  
- `Callable#call()`：有返回值，可以抛受检异常，常配合 `Future` 获取执行结果。

【延伸考点】
- `Executors.callable(Runnable)` 可把 `Runnable` 适配为 `Callable`。  
- 需要返回值或异常传播时优先用 `Callable`。

【问题】13. 概括解释线程的几种可用状态。

【参考答案】

- 概念层面可理解为：就绪（可运行）、运行、阻塞/等待、计时等待（sleep/超时等待）、终止。  

【延伸考点】
- `Thread.State` 枚举：`NEW`、`RUNNABLE`、`BLOCKED`、`WAITING`、`TIMED_WAITING`、`TERMINATED`。  
- 用 `jstack`/`jcmd` 排查线程状态与锁等待。

【问题】14. 同步方法和同步代码块的区别是什么？

【参考答案】

- 同步方法：`synchronized` 修饰方法。  
  - 实例方法默认锁 `this`；静态同步方法锁 `Class` 对象。  
- 同步代码块：`synchronized(lock) { ... }`，可自定义锁对象，粒度更细，只保护关键临界区。

【延伸考点】
- 尽量缩小锁范围，避免把 I/O、RPC 等慢操作放在锁内。  
- 显式锁 `ReentrantLock` 提供更丰富能力（可中断、公平锁、条件队列等）。

【问题】15. 在监视器（Monitor）内部，是如何做线程同步的？

【参考答案】

- Java 的 `synchronized` 基于对象监视器（Monitor）：同一把锁同一时刻只允许一个线程进入临界区；  
- 进入同步块/方法需要获取锁，退出时释放锁，其他线程阻塞等待。

【延伸考点】
- `wait/notify/notifyAll` 必须在持有监视器锁时调用（在 `synchronized` 内）。  
- JVM 对锁有多种优化与状态（偏向锁/轻量级/重量级等，了解即可）。

【问题】程序计数器为什么是私有的？

【参考答案】

- 程序计数器用于记录当前线程执行到的字节码位置：  
  - 解释器依赖它实现顺序/分支/循环/异常处理等流程控制；  
  - 多线程切换时可恢复到正确执行位置。  
- 因为每个线程执行路径不同，所以程序计数器必须是**线程私有**的。

【延伸考点】
- JVM 运行时数据区的线程私有/共享：程序计数器、虚拟机栈、本地方法栈是私有；堆、方法区是共享。

【问题】并发编程的三个重要特性？

【参考答案】

- **原子性**：操作不可分割，要么全部执行要么不执行；`synchronized`、锁、原子类可保证。  
- **可见性**：一个线程修改共享变量，其他线程能及时看到；`volatile`、锁可保证。  
- **有序性**：避免重排序导致的执行顺序问题；`volatile`、锁、happens-before 规则可约束。

【延伸考点】
- JMM 的 happens-before 规则与 `volatile` 的内存语义。  
- 双重检查锁（DCL）必须配合 `volatile` 的原因。

【问题】常见的线程安全类有哪些？

【参考答案】

- 早期同步容器/工具：`Vector`、`Hashtable`、`Stack`、`StringBuffer`、`Timer/TimerTask` 等（多数已不推荐作为首选）。  
- 原子类：`AtomicInteger`、`AtomicLong`、`AtomicReference` 等（基于 CAS）。  
- 阻塞队列：`ArrayBlockingQueue`、`LinkedBlockingQueue`、`PriorityBlockingQueue`、`LinkedBlockingDeque` 等。  
- 并发集合：`ConcurrentHashMap`、`ConcurrentSkipListMap/Set`、`CopyOnWriteArrayList/Set` 等。  
- 包装器：`Collections.synchronizedXxx(...)` 系列。

【延伸考点】
- `ConcurrentHashMap` 与 `Collections.synchronizedMap` 的差异（锁粒度与吞吐）。  
- `CopyOnWriteArrayList` 适合读多写少，写入会复制数组，内存与写成本较高。

### 死锁

【问题】16. 什么是死锁（deadlock）？

【参考答案】

- 死锁是指：多个线程/进程在执行过程中，因为**争夺资源**或**通信**造成彼此等待，导致所有相关线程都无法继续执行的一种状态。  
- 典型表现是：线程 A 持有资源 1 等待资源 2；线程 B 持有资源 2 等待资源 1，形成循环等待，程序卡死。

【延伸考点】
- 产生死锁的四个必要条件：互斥、占有且等待、不可剥夺、循环等待。  
- 常见预防手段：固定加锁顺序、超时获取锁（`tryLock`）、减少锁粒度、避免嵌套锁等。

【问题】请手写一个简单的死锁示例，并说明原因。

【参考答案（示例代码）】

```java
public class DeadlockTest {
    private static final Object A = new Object();
    private static final Object B = new Object();

    public static void main(String[] args) {
        new Thread(DeadlockTest::lockAThenB, "T1").start();
        new Thread(DeadlockTest::lockBThenA, "T2").start();
    }

    static void lockAThenB() {
        synchronized (A) {
            System.out.println(Thread.currentThread().getName() + " 获取到 A 锁");
            try {
                Thread.sleep(500);
            } catch (InterruptedException ignored) {}
            synchronized (B) {
                System.out.println(Thread.currentThread().getName() + " 获取到 B 锁");
            }
        }
    }

    static void lockBThenA() {
        synchronized (B) {
            System.out.println(Thread.currentThread().getName() + " 获取到 B 锁");
            try {
                Thread.sleep(500);
            } catch (InterruptedException ignored) {}
            synchronized (A) {
                System.out.println(Thread.currentThread().getName() + " 获取到 A 锁");
            }
        }
    }
}
```

- 线程 T1：先持有 A 再请求 B；线程 T2：先持有 B 再请求 A；  
- 当 T1 拿到 A、T2 拿到 B 后，双方都在等待对方释放资源，形成循环等待，产生死锁。

【延伸考点】
- 调试死锁：使用 `jstack` 或 IDE 线程分析工具查看线程堆栈与锁等待。  
- 尽量避免嵌套锁，或在设计时统一加锁顺序。

【问题】17. 如何确保 N 个线程可以访问 N 个资源同时又不导致死锁？

【参考答案】

- 关键在于**打破死锁的四个必要条件之一**，常用策略包括：  
  - 固定获取锁的顺序：所有线程按照统一顺序获取资源（如总是先锁 A 再锁 B），避免循环等待；  
  - 一次性申请所有资源：避免“占有且等待”；  
  - 获取不到全部资源时，主动释放已持有资源并稍后重试；  
  - 使用带超时的锁获取（如 `tryLock`），在超时后放弃并重试。

【延伸考点】
- 数据库中对死锁的检测与回滚机制（如 InnoDB 的死锁检测）。  
- 使用更高层抽象（如任务队列、Actor 模型）来降低显式加锁的频率。

【问题】操作系统中死锁的四个必要条件是什么？

【参考答案】

1）**互斥条件**：  
- 某资源在一段时间内只能被一个进程/线程占用，不能共享使用。

2）**请求和保持条件**：  
- 进程已至少持有一个资源，但又提出新的资源请求；  
- 新资源被其他进程占用时，请求被阻塞，但进程保持已占有的资源不释放。

3）**不可抢占条件**：  
- 资源一旦分配给进程，在其使用完之前，不能被强行剥夺，只能由进程主动释放。

4）**循环等待条件**：  
- 存在一个进程-资源的循环链，每个进程都占有下一个进程所需的资源，同时又等待上一个进程释放资源。

只要破坏其中任意一个条件，就可以防止死锁的发生。





### 同步与异步

【问题】线程同步有哪些常见机制？

【参考答案】

- **临界区**：  
  - 同一时间只允许一个线程进入的代码区域，用于保护共享资源；  
  - Java 中对应的就是 `synchronized`、显式锁等机制。
- **互斥量（Mutex）**：  
  - 一种互斥锁概念，可用于线程间或进程间同步；  
  - 在 Java 中一般由 `ReentrantLock` 等实现。
- **事件 / 条件变量**：  
  - 用于线程间通知与协作，等待某个条件满足再继续执行；  
  - Java 中可以类比 `Object.wait/notify`、`Condition`。
- **信号量（Semaphore）**：  
  - 控制同时访问某资源的线程数量（计数信号量）；  
  - 对应 Java 的 `java.util.concurrent.Semaphore`。

【延伸考点】
- 了解 Java 同步原语：内置锁、`ReentrantLock`、`Semaphore`、`CountDownLatch`、`CyclicBarrier` 等。  
- 正确选择同步机制以平衡性能与复杂度。

【问题】同步（Sync）和异步（Async）的区别？

【参考答案】

- **同步调用**：  
  - 发出调用后，调用方在结果返回前一直等待，不能继续后续操作；  
  - 调用方与被调用方在“时间上强耦合”。
- **异步调用**：  
  - 发出调用后，调用方可以立即返回，继续处理其他事情；  
  - 被调用方完成任务后，通过回调、事件、Future 等方式通知结果。

【延伸考点】
- 异步 + 非阻塞 I/O 在高并发系统中的应用（如 Netty、Reactor 模式）。  
- 不要把“同步/异步”与“阻塞/非阻塞”混为一谈，它们是两个维度。

【问题】阻塞和非阻塞的区别？

【参考答案】

- **阻塞调用**：  
  - 调用结果返回前，当前线程会被挂起，直到条件满足或超时；  
  - 如传统 `read()` 阻塞等待数据、`Thread.sleep()` 等。
- **非阻塞调用**：  
  - 调用立即返回，如果条件未满足通过状态码或异常告知；  
  - 调用方可以轮询或搭配回调、事件机制继续处理。

【延伸考点】
- 阻塞/非阻塞更多指 I/O 或系统调用的行为；同步/异步指**编程模型与调用者的等待策略**。  



### 唤醒与阻塞

【问题】sleep() 和 wait() 的区别与共同点是什么？

【参考答案】

共同点：  
- 都会让当前线程“暂停执行”一段时间或直到被唤醒。

主要区别：  
- `sleep(long)` 是 `Thread` 的静态方法，只会让线程休眠，不会释放持有的锁；时间到或被中断后自动苏醒。  
- `wait()` 是 `Object` 的实例方法，必须在同步块/方法内调用；调用后会**释放当前持有的锁**，线程进入等待队列，直到被 `notify/notifyAll` 或超时唤醒。  
- `wait()` 主要用于线程间通信；`sleep()` 通常用于简单的时间控制。

【延伸考点】
- `wait(long timeout)` 和 `sleep(long)` 在中断/超时行为上的差异。  
- 为什么 `wait/notify` 属于 `Object` 而不是 `Thread`。

【问题】启动线程为什么调用 start() 而不是直接调用 run()？

【参考答案】

- `start()`：  
  - 通知 JVM 创建新的操作系统级线程，并让该线程进入就绪/运行状态；  
  - JVM 在新线程中自动调用该 `Thread` 对象的 `run()` 方法，是真正的并发执行。
- 直接调用 `run()`：  
  - 只是在**当前线程**中执行一个普通方法调用；  
  - 不会创建新线程，不会产生并发。

【延伸考点】
- `Thread` 生命周期：NEW → RUNNABLE → BLOCKED/WAITING/TIMED_WAITING → TERMINATED。  
- 同一个 `Thread` 对象的 `start()` 只能调用一次。

【问题】线程阻塞常见由哪些方法导致？如何解除阻塞？

【参考答案】

- `Thread.sleep(ms)`：  
  - 让当前线程休眠指定时间，期间不释放锁；  
  - 时间到或被 `interrupt()` 中断后恢复到可运行状态。
- `Object.wait()`：  
  - 在线程持有对象监视器锁下调用，释放锁并进入等待队列；  
  - 通过 `notify/notifyAll` 或超时唤醒，再次尝试获得锁后恢复执行。
- `Thread.join()`：  
  - 当前线程等待另一个线程执行结束后再继续。  
- I/O 阻塞：  
  - 在系统调用/网络 I/O 等处阻塞，解除由操作系统控制，Java 代码难以直接干预。
- `Thread.yield()`：  
  - 提示调度器当前线程可以让出 CPU，但不进入阻塞状态，只是“让出时间片”的一种建议。

【延伸考点】
- 区分“阻塞状态”和“可运行但未获得时间片”的状态。  
- `LockSupport.park/unpark` 在 J.U.C 中的作用。

【问题】什么是上下文切换（Context Switch）？

【参考答案】

- 当 CPU 从一个线程/进程切换到另一个时，需要保存当前执行单元的寄存器、程序计数器等状态，并恢复另一个执行单元的状态；  
- 这个“保存现场 + 恢复现场”的过程就称为一次上下文切换；  
- 过于频繁的上下文切换会带来性能损耗。

【延伸考点】
- 过多线程、锁竞争、过细粒度任务可能导致上下文切换频繁。  
- 使用合适的线程池大小、减少不必要的线程创建有助于降低切换成本。





### 线程池

【问题】为什么要使用线程池？

【参考答案】

- **降低资源消耗**：重用已创建线程，避免频繁创建/销毁带来的开销。  
- **提高响应速度**：任务到达时若有空闲线程，可立即执行，无需等待新线程创建。  
- **提高可管理性**：统一管理线程数量、队列长度和拒绝策略，避免无节制创建线程导致内存/CPU 压力。

【延伸考点】
- 在 Web 服务中，线程池大小与 CPU 核数、I/O 比例的关系。  
- 饱和策略（`AbortPolicy`、`CallerRunsPolicy` 等）的选择。

【问题】线程池中 execute() 和 submit() 的区别是什么？

【参考答案】

- `execute(Runnable)`：  
  - 只接受 `Runnable` 任务；  
  - 无返回值，无法直接获取任务执行结果；  
  - 异常会被线程池的 UncaughtExceptionHandler 处理或直接打印日志。
- `submit(Runnable/Callable)`：  
  - 同时支持 `Runnable` 和 `Callable`；  
  - 返回 `Future`，可以判断任务是否完成、获取返回值或异常；  
  - 对于异常，可在 `future.get()` 时获取并处理。

【延伸考点】
- `submit(Runnable)` 返回的 `Future.get()` 通常是 `null`，但可用于感知任务是否成功或异常。  
- 在线程池中推荐统一通过 `submit + Future` 或显式异常处理，避免“静默失败”。

【问题】如何优雅地终止线程池？

【参考答案】

- 使用 `shutdown()`：  
  - 停止接收新任务；  
  - 等待已提交的任务（包括队列中）都执行完再关闭线程池；  
  - 是默认推荐的“平滑关闭”方式。
- 使用 `shutdownNow()`：  
  - 停止接收新任务；  
  - 尝试中断正在执行的任务，并返回未执行的队列任务列表；  
  - 需要任务正确响应中断，否则可能无法马上结束。

【延伸考点】
- 结合 `awaitTermination` 等待一段时间，超时后再选择是否 `shutdownNow()`。  
- 设计任务代码时要定期检查中断状态，支持可中断的阻塞调用。



【问题】如何创建线程池？为什么不推荐直接用 Executors？

【参考答案】

- 《阿里巴巴 Java 开发手册》建议：**不要直接使用 `Executors` 创建线程池**，而是显式使用 `ThreadPoolExecutor`，因为这样能更明确地控制线程数、队列大小、拒绝策略，避免资源耗尽。  
- `Executors` 常见风险：  
  - `newFixedThreadPool` / `newSingleThreadExecutor`：`workQueue` 可能接近无界（`Integer.MAX_VALUE`），任务堆积导致 OOM；  
  - `newCachedThreadPool` / `newScheduledThreadPool`：线程数可能接近无界（`Integer.MAX_VALUE`），创建大量线程导致 OOM。

【延伸考点】
- 线程池核心三要素：线程数、队列、拒绝策略。  
- 建议在线上系统对线程池参数做可配置与监控（活跃线程数、队列长度、拒绝次数）。

【问题】ThreadPoolExecutor 构造函数重要参数有哪些？分别代表什么？

【参考答案】

- `corePoolSize`：核心线程数，线程池最少保持的线程数量。  
- `maximumPoolSize`：最大线程数，当队列满时，最多能扩展到的线程数量。  
- `workQueue`：任务队列，核心线程满时新任务进入队列等待。  
- `keepAliveTime`：非核心线程空闲存活时间，超过则回收（部分配置允许核心线程也超时回收）。  
- `unit`：`keepAliveTime` 时间单位。  
- `threadFactory`：创建线程的工厂（线程命名、优先级、是否守护线程等）。  
- `handler`：拒绝策略（饱和策略）。

【延伸考点】
- 线程池的任务提交流程：核心线程 → 入队 → 非核心线程扩容 → 拒绝策略。  
- `allowCoreThreadTimeOut(true)` 的使用场景。

【问题】ThreadPoolExecutor 的拒绝策略有哪些？

【参考答案】

- `AbortPolicy`：抛 `RejectedExecutionException`（默认）。  
- `CallerRunsPolicy`：由提交任务的线程自己执行任务（降低提交速率，但可能影响调用线程响应）。  
- `DiscardPolicy`：直接丢弃任务，不抛异常。  
- `DiscardOldestPolicy`：丢弃队列中最旧的任务，然后尝试重新提交当前任务。

【延伸考点】
- 线上系统通常需要记录拒绝次数与任务丢弃情况，避免“悄悄丢任务”。  
- 拒绝策略应结合业务是否允许丢弃、是否可降级来选择。

【问题】如何合理设置线程池参数？

【参考答案】

- 先区分任务类型：  
  - **CPU 密集型**：主要消耗 CPU 计算，如排序、加解密等；  
  - **I/O 密集型**：大量时间等待 I/O，如网络请求、磁盘读写、DB 等。
- 常用经验：  
  - CPU 密集型：线程数 \(\approx N_{\text{CPU}} + 1\)；  
  - I/O 密集型：线程数一般可更大（例如 \(\approx 2N_{\text{CPU}}\) 作为起点），最终以压测与监控结果为准。
- 获取 CPU 核数：`Runtime.getRuntime().availableProcessors()`。

【延伸考点】
- 更通用的估算：\(N_{\text{threads}} \approx N_{\text{CPU}} \times (1 + \frac{W}{C})\)，其中 \(W\) 是等待时间、\(C\) 是计算时间。  
- 生产实践：线程池参数可配置 + 监控报警 + 压测验证。

【问题】线程池中的阻塞队列有哪些？各自特点是什么？

【参考答案】

- `ArrayBlockingQueue`：  
  - 基于数组的**有界**阻塞队列，FIFO；  
  - 可选择公平/非公平锁，公平会降低吞吐。
- `LinkedBlockingQueue`：  
  - 基于链表的阻塞队列，FIFO；  
  - 常见实现默认容量很大，容易造成任务堆积（需显式设置容量）。
- `SynchronousQueue`：  
  - 不存储元素，`put` 必须等待 `take`，适合“直接移交”场景；  
  - 常用于 `newCachedThreadPool`。
- `PriorityBlockingQueue`：  
  - 支持优先级的无界阻塞队列；  
  - 元素需要可比较或提供 `Comparator`。

【延伸考点】
- 无界队列会把压力转移到内存；有界队列能更早触发拒绝策略实现限流/降级。  
- 业务场景决定队列选择：延迟敏感 vs 吞吐优先 vs 需要优先级。





### 原子类CAS

【问题】介绍一下 Atomic 原子类？

【参考答案】

- Atomic 原子类用于在并发场景下对变量进行**原子更新**，常见实现基于 **CAS + volatile + CPU 原子指令**，在很多场景下可以替代 `synchronized` 降低锁竞争开销。  
- 常见分类：  
  - 基本类型：`AtomicInteger`、`AtomicLong`、`AtomicBoolean`；  
  - 数组类型：`AtomicIntegerArray`、`AtomicLongArray`、`AtomicReferenceArray`；  
  - 引用类型：`AtomicReference`、`AtomicStampedReference`、`AtomicMarkableReference`；  
  - 字段更新器：`AtomicIntegerFieldUpdater`、`AtomicLongFieldUpdater`、`AtomicReferenceFieldUpdater`。

【延伸考点】
- 原子类适合“竞争不激烈、临界区很小”的计数/状态更新；高竞争下 CAS 自旋可能变慢。  
- `LongAdder`/`LongAccumulator` 在高并发计数场景通常比 `AtomicLong` 更好。

【问题】介绍下 AtomicInteger 的使用与常见方法？

【参考答案】

常用方法：
- `get()`：获取当前值  
- `set(newValue)`：直接设置  
- `getAndSet(newValue)`：返回旧值并设置新值  
- `incrementAndGet()` / `getAndIncrement()`：自增（返回新值/旧值）  
- `decrementAndGet()` / `getAndDecrement()`：自减  
- `addAndGet(delta)` / `getAndAdd(delta)`：加指定增量  
- `compareAndSet(expect, update)`：CAS 更新  
- `lazySet(newValue)`：最终会设置成功，但可能短时间内对其他线程不可见（弱内存语义）

示例：

```java
class AtomicIntegerTest {
    private final AtomicInteger count = new AtomicInteger();

    public void increment() {
        count.incrementAndGet();
    }

    public int getCount() {
        return count.get();
    }
}
```

【延伸考点】
- CAS 的成功依赖“期望值与当前值相等”，失败则需要重试（自旋）。  
- `lazySet` 常用于写多读少且允许短暂可见性延迟的场景（了解即可）。

【问题】AtomicInteger 的 getAndIncrement 用到 CAS，原理是什么？

【参考答案】

- CAS：Compare-And-Swap（比较并交换），包含 3 个核心值：  
  - V：内存中的当前值  
  - A：期望值（旧值）  
  - B：新值  
- 当且仅当 `V == A` 时，将 `V` 原子更新为 `B`；否则更新失败（返回 false 或返回旧值），调用方通常会重试直到成功。

【延伸考点】
- `Unsafe`/VarHandle 最终会映射到 CPU 的原子指令（如 x86 的 `cmpxchg`）。  
- CAS 是无锁并发的重要基础，但不是“没有开销”。

【问题】CAS 会带来什么问题？

【参考答案】

- **自旋开销**：竞争激烈时 CAS 失败率高，会导致循环重试、CPU 空转消耗大；  
- **只能保证单变量原子性**：多个共享变量需要整体一致性时，单纯 CAS 难以保证，需要锁或更高级原语；  
- **ABA 问题**：值从 A→B→A，CAS 可能误认为“没变”。

【延伸考点】
- 对高并发热点计数用 `LongAdder` 降低竞争。  
- 对多变量一致性场景更适合用锁、事务或不可变对象与复制更新策略。

【问题】什么是 ABA 问题？如何解决？

【参考答案】

- ABA 问题：CAS 只比较“当前值是否等于期望值”。如果一个值从 A 变成 B 又变回 A，CAS 会认为它没有变化，但实际上发生过变化。  
- 解决思路：给数据增加“版本号/时间戳”，让 A→B→A 变成 1A→2B→3A。  
- Java 中常用：`AtomicStampedReference`（版本号 stamp）或 `AtomicMarkableReference`（标记位）。

【延伸考点】
- ABA 是否是问题取决于业务语义：有些场景允许 ABA（只关心最终值），有些场景不允许（涉及资源回收、指针复用等）。

1.编译器优化重排：编译器在不改变单线程程序语义的前提下，可以重新安排语句的执行顺序。
2.指令级的并行重排：现代处理器采用了指令级并行技术来将多条指令重叠执行。如果不存在数据依赖性，处理器可以改变语句对应机器指令的执行顺序。
3.内存系统的重排：由于处理器使用缓存和读/写缓冲区，这使得加载和存储操作看上去可能是在乱序执行。

如果线程2 num=num+5 在线程1设置num=1之前执行，那么线程2的num变量值为5

修改为：static volatile boolean flag = false;
原理：在volatile生成的指令序列前后插入内存屏障（Memory Barries）来禁止处理器重排序。
volatile写的场景如何插入内存屏障：
在每个volatile写操作的前面插入一个StoreStore屏障（写-写 屏障）。
在每个volatile写操作的后面插入一个StoreLoad屏障（写-读 屏障）。
StoreStore屏障可以保证在volatile写（flag赋值操作flag=true）之前，其前面的所有普通写（num的赋值操作num=1) 操作已经对任意处理器可见了，保障所有普通写在volatile写之前刷新到主内存。

volatile读场景如何插入内存屏障：
在每个volatile读操作的后面插入一个LoadLoad屏障（读-读 屏障）。
在每个volatile读操作的后面插入一个LoadStore屏障（读-写 屏障）。
LoadStore屏障可以保证其后面的所有普通写（num的赋值操作num=num+5) 操作必须在volatile读（if(flag)）之后执行。

### AQS

```
介绍一下AQS？

AQS的全称为（AbstractQueuedSynchronizer），这个类在java.util.concurrent.locks包下面。
AQS是一个用来构建锁和同步器的框架，使用AQS能简单且高效地构造出应用广泛的大量的同步器，比如我们提到的ReentrantLock，Semaphore，其他的诸如ReentrantReadWriteLock，SynchronousQueue，FutureTask等等皆是基于AQS的。
```

```
说一下对AQS原理的理解？

AQS核心思想是，如果被请求的共享资源空闲，则将当前请求资源的线程设置为有效的工作线程，并且将共享资源设置为锁定状态。如果被请求的共享资源被占用，那么就需要一套线程阻塞等待以及被唤醒时锁分配的机制，这个机制AQS是用CLH队列锁实现的，即将暂时获取不到锁的线程加入到队列中。

AQS 内部数据和方法，可以简单拆分为：
1：一个 volatile 的整数成员表示同步状态，同时提供了 setState 和 getState 方法。AQS 使用 CAS 对该同步状态进行原子操作实现对其值的修改。
2：一个先入先出（FIFO）的等待线程队列，以实现多线程间竞争和等待，这是 AQS 机制的核心之一。CLH(Craig,Landin,and Hagersten)队列是一个虚拟的双向队列（虚拟的双向队列即不存在队列实例，仅存在结点之间的关联关系）。AQS是将每条请求共享资源的线程封装成一个CLH锁队列的一个结点（Node）来实现锁的分配。
3：各种基于 CAS 的基础操作方法，以及各种期望具体同步结构去实现的 acquire/release 方法。

利用 AQS 实现一个同步结构，至少要实现两个基本类型的方法，分别是 acquire 操作，获取资源的独占权；还有就是 release 操作，释放对某个资源的独占。
```

```
AQS组件有哪几个？

Semaphore(信号量)-允许多个线程同时访问： synchronized 和 ReentrantLock 都是一次只允许一个线程访问某个资源，Semaphore(信号量)可以指定多个线程同时访问某个资源。

CountDownLatch （倒计时器）： CountDownLatch 是一个同步工具类，用来协调多个线程之间的同步。这个工具通常用来控制线程等待，它可以让某一个线程等待直到倒计时结束，再开始执行。

CyclicBarrier(循环栅栏)： CyclicBarrier 和 CountDownLatch 非常类似，它也可以实现线程间的技术等待，但是它的功能比 CountDownLatch 更加复杂和强大。主要应用场景和 CountDownLatch 类似。CyclicBarrier 的字面意思是可循环使用（Cyclic）的屏障（Barrier）。它要做的事情是，让一组线程到达一个屏障（也可以叫同步点）时被阻塞，直到最后一个线程到达屏障时，屏障才会开门，所有被屏障拦截的线程才会继续干活。CyclicBarrier 默认的构造方法是 CyclicBarrier(int parties)，其参数表示屏障拦截的线程数量，每个线程调用 await() 方法告诉 CyclicBarrier 我已经到达了屏障，然后当前线程被阻塞。

### synchronized

```
谈谈synchronized关键字以及使用？底层原理是什么？

synchronized关键字解决的是多个线程之间访问资源的同步性，synchronized关键字可以保证被它修饰的方法或者代码块在任意时刻只能有一个线程执行。
synchronized属于重量级锁，效率低下，因为监视器锁（monitor）是依赖于底层的操作系统的 Mutex Lock 来实现的，Java 的线程是映射到操作系统的原生线程之上的。
synchronized 关键字加到 static 静态方法和 synchronized(class)代码块上都是是给 Class 类上锁。synchronized 关键字加到实例方法上是给对象实例上锁。尽量不要使用 synchronized(String a) 因为JVM中，字符串常量池具有缓存功能！

synchronized 关键字底层原理属于 JVM 层面。
synchronized 同步语句块的实现使用的是 monitorenter 和 monitorexit 指令，其中 monitorenter 指令指向同步代码块的开始位置，monitorexit 指令则指明同步代码块的结束位置。 当执行 monitorenter 指令时，线程试图获取锁也就是获取 monitor(monitor对象存在于每个Java对象的对象头中，synchronized 锁便是通过这种方式获取锁的，也是为什么Java中任意对象可以作为锁的原因) 的持有权。当计数器为0则可以成功获取，获取后将锁计数器设为1也就是加1。相应的在执行 monitorexit 指令后，将锁计数器设为0，表明锁被释放。如果获取对象锁失败，那当前线程就要阻塞等待，直到锁被另外一个线程释放为止。

public class Singleton {
	//uniqueInstance 采用 volatile 关键字修饰也是很有必要的,可以禁止JVM指令重排序
    private volatile static Singleton uniqueInstance;

    private Singleton() {
    }

    public  static Singleton getUniqueInstance() {
       //先判断对象是否已经实例过，没有实例化过才进入加锁代码
        if (uniqueInstance == null) {
            //类对象加锁
            synchronized (Singleton.class) {
                if (uniqueInstance == null) {
                    uniqueInstance = new Singleton();
                }
            }
        }
        return uniqueInstance;
    }
}

synchronized和ReentrantLock 的区别?

两者都是可重入锁:自己可以再次获取自己的内部锁。比如一个线程获得了某个对象的锁，此时这个对象锁还没有释放，当其再次想要获取这个对象的锁的时候还是可以获取的，如果不可锁重入的话，就会造成死锁。同一个线程每次获取锁，锁的计数器都自增1，所以要等到锁的计数器下降为0时才能释放锁。

synchronized 依赖于 JVM 而 ReentrantLock 依赖于 API

ReentrantLock 比 synchronized 增加了一些高级功能:①等待可中断；②可实现公平锁；③可实现选择性通知（锁可以绑定多个条件）

```
synchronized和volatile的区别？

synchronized 关键字和 volatile 关键字是两个互补的存在，而不是对立的存在：
volatile关键字是线程同步的轻量级实现，所以volatile性能肯定比synchronized关键字要好。
volatile只能修饰实例变量和类变量，synchronized可以修饰方法和代码块。
多线程访问volatile关键字不会发生阻塞，而synchronized关键字可能会发生阻塞
volatile关键字能保证数据的可见性，但不能保证数据的原子性。synchronized关键字两者都能保证。
volatile关键字主要用于解决变量在多个线程之间的可见性，而 synchronized关键字解决的是多个线程之间访问资源的同步性。

```
1. 面试官：Synchronized有用过吗？谈谈你对它的理解
（2）回答：Synchronized是Java的关键词，JVM实现的一种可以实现并发产生的多个线程互斥同步访问共享资源的方式，也可以说是一种 “同步互斥锁”，在实际代码中可用于修饰代码块、方法、静态方法以及类；适用于单体应用系统架构



2.面试官：嗯，说一说它的原理？
（2）回答： 通过查看被Synchronized 修饰过的代码块编译后的字节码，会发现编译器会在被Synchronized修饰过的代码块的前、后生成两个字节码指令：monitorenter、monitorexit；
这两个字节码指令的含义：当JVM执行到monitorenter指令时，首先会尝试着先获取对象（共享资源）的锁，如果该对象没有被锁定、又或者当前线程已经拥有了这个对象的锁时，则锁的计数器count加1，即执行 +1 操作；当JVM执行monitorexit指令时，则将锁的计数器count减一，即执行 -1 操作；当计数器count为0时 ，该对象的锁就被释放了！！
如果当前线程获取该对象的锁失败了，则进入堵塞等待状态，直到该对象的锁被另外一个线程释放为止；即Java中的Synchronize底层其实是通过对象（共享资源）头、尾设置标记，从而实现锁的获取和释放。



3.面试官：你刚才提到获取对象的锁，说一说“锁”到底是什么，如何确定对象的锁？
（2）回答： “锁” 可以理解为monitorenter和monitorexit字节码指令之间的一个 Reference类型的参数，即要锁定Lock和解锁UnLock的对象
众所周知，使用Synchronized可以修饰不同的对象，因此，对应的对象的锁可以这么确定：
A.如果Synchronized 明确指定了“锁”的对象，比如Synchronized变量、 Synchronized(this) 等，说明加、解锁的即为该变量、当前对象；
B.若 Synchronized 修饰的方法为非静态方法，表示此方法对应的对象为“锁”对象； 若 Synchronized 修饰的方法为静态方法，则表示此方法对应的类对象为“锁”对象；
注意：当一个对象被锁住时，对象里面所有用Synchronized 修饰的方法都将产生堵塞， 而对象里非Synchronized 修饰的方法可正常被调用，不受锁的影响；




4.面试官：什么叫可重入锁，为什么说Synchronized是可重入锁？
（2）回答：通俗地讲，“可重入”指的是：当 当前线程获取到了当前对象的锁之后，如果后续的操作仍然需要获取获取该对象的锁时，可以不用再次重新获取，即可以直接操作该对象（共享资源）；
可重入性是锁的一个基本要求，是为了解决自己锁死自己的情况，比如一个类的同步方法调用另一个同步方法时，假如Synchronized不支持重入，进入method2方法时当前线程已经获得锁，而在method2方法里面执行method1时当前线程又要去尝试获取锁，这时如果不支持重入，它就要等待释放，把自己阻塞，导致很有可能自己锁死自己！
对Synchronized来说，可重入性是显而易见的，刚才提到，在执行monitorenter指令时，如果这个对象没有锁定，或者当前线程已经拥有了这个对象的锁，就把锁的计数器+1，其实本质上就是通过这种方式实现了可重入性（而不是已拥有了锁则不能继续获取）。




5.面试官：说一说JVM底层对Java的原生锁做了哪些优化？
（2）回答：在Java 6以前，Monitor的实现完全依赖底层操作系统的互斥锁来实现，也就是上面在问题2中所阐述的获取、释放锁的逻辑；由于Java的线程与操作系统的原生线程有映射关系，如果要将一个线程进行阻塞或唤起 都需要操作系统的协助，这就需要从用户态切换到内核态来执行，这种切换代价十分昂贵，很耗处理器时间，现代JDK中做了大量的优化；
一种优化是使用自旋锁，即在线程进行阻塞操作之前先让线程自旋等待一段时间，可能在等待期间其他线程已经解锁，这时就无需再让线程执行阻塞操作，避免了用户态到内核态的切换。
而现代JDK中还提供了三种不同的Monitor实现，也就是三种不同的锁：偏向锁、轻量级锁、重量级锁
这三种锁使得JDK得以优化Synchronized的运行，当JVM检测到不同的竞争状况时，会自动切换到适合的锁实现，这就是锁的升级、降级。
当没有竞争出现时，默认使用偏向锁，锁对象的对象头中记录一下当前获取到该锁的线程ID，该线程下次如果又来获取该锁就可以直接获取到了，也就是支持锁重入
当两个或以上线程交替获取锁，但并没有在对象上并发的获取锁时，偏向锁升级为轻量级锁。在此阶段，线程采取CAS的自旋方式尝试获取锁，避免阻塞线程造成的cpu在用户态和内核态间转换的消耗
两个或以上线程并发的在同一个对象上进行同步时，为了避免无用CAS消耗CPU，轻量级锁会升级为重量级锁





6.面试官：嗯，不错，Synchronized是公平锁还是非公平锁，为什么？
（2）回答：非公平；非公平主要表现在获取锁的行为上：并非是按照申请锁的时间前后给等待线程分配锁的，每当锁被释放后，任何一个线程都有机会竞争到锁，这样做的目的是为了提高执行性能，缺点是可能会产生线程饥饿现象；




7.面试官： 为什么说Synchronized是悲观锁？
（2）回答：因为Synchronized的并发策略是悲观的：即不管是否会产生竞争，任何的数据操作都必须要加锁，包括“从用户态切换到核心态”、“维护锁计数器”和“检查被阻塞的线程是否需要被唤醒”等操作；
随着硬件指令集的发展，我们可以使用基于冲突检测的乐观并发策略，即先进行操作，如果没有其他线程征用数据，那操作就成功了；
如果共享数据有征用，产生了冲突，那就再进行其他的补偿措施，这种乐观的并发策略的许多实现不需要线程挂起，所以被称为非阻塞同步。




8.面试官： 那你了解乐观锁吗，它的实现原理又是什么，能讲讲吗？
（2）回答：乐观锁，顾名思义表示系统总是认为当前的并发情况是乐观的，而不需要通过加各种锁进行控制；
乐观锁的实现原理是CAS机制（Compare And Swap，比较并交换），一种在JUC中广泛使用的算法；它涉及到三个操作数：内存值V、预期值A、新值B，当且仅当预期值A和内存值V相等时才将内存值V修改为新值B；
其底层实现逻辑：首先检查某块内存的值是否跟之前我读取的是一样的，如果不一样则表示期间此内存值已经被别的线程更改过，舍弃本次操作，否则说明期间没有其他线程对此内存值操作，可以把新值设置给此块内存，即间接意味着获取锁成功！
CAS具有原子性，它的原子性是由CPU硬件指令实现保证的，即通过JNI调用Native方法，从而调用由C++编写的硬件级别指令，JDK中提供了Unsafe类来执行这些操作（查看JUC很多类的底层源码会发现 Unsafe.compareAndSwapxxx() 的调用无处不在，很牛逼！！！）



9.面试官：那乐观锁就一定是好的吗？
（2）回答：乐观锁可以避免 悲观锁独占对象这一现象 的出现，同时也提高了并发性能，但它也有一些缺点：
A.  乐观锁只能保证一个共享变量的原子操作：如果多一个或几个变量，乐观锁将变得力不从心，但互斥锁能轻易解决，不管对象数量多少及对象颗粒度大小；
B.  长时间自旋可能导致开销大。假如CAS长时间不成功而一直自旋，会给CPU带来很大的开销；
C.  ABA问题：CAS的核心思想是通过比对内存值与预期值是否一样而判断内存值是否被改过，但这个判断逻辑不够严谨；
假如内存值原来是A，后来被一线程改为B，最后又被改回了A，则CAS认为此内存值并没有发生改变，但实际上是有被其他线程改过的，这种情况对依赖过程值的情景的运算结果影响很大。




10.面试官：刚提到ABA的问题，那有什么办法解决吗？
（2）回答：解决的思路是引入版本号，每次变量更新时都把版本号加1，同时如果条件允许，还需要额外建立数据更新历史表，并同时维护好版本号version 和 数据变更记录的映射关系！




11.面试官：跟Synchronized相比，可重入锁ReentrantLock的实现原理有什么不同？
（2）回答：其实，几乎所有锁的实现原理都是为了达到同个目的：让所有的线程都能看到某种标记，同一时刻只能有一个线程获取到锁；
Synchronized通过在对象头中设置标记MarkWord实现了这一目的，是一种JVM原生的锁实现方式；
而ReentrantLock以及所有的基于Lock接口的实现类，则是通过一个volitile关键字修饰的int类型变量，并保证每个线程都能拥有对该int变量的可见性和原子性，其本质是基于所谓的AQS框架；




12.面试官：你刚刚提到了AQS，那你说说AQS的实现原理？
（2）回答：AQS，即 AbstractQueuedSynchronizer  抽象队列同步器，是一个用来构建锁和同步器的类，JUC  Lock包下的锁（常用的有ReentrantLock、ReadWriteLock），以及其他的像Semaphore、CountDownLatch，甚至是早期的FutureTask等，都是基于AQS来构建的；
A：AQS在内部定义了一个变量：volatile int state，用于表示同步状态：当线程调用lock方法时，如果state=0，说明没有任何线程占有共享资源的锁，可以获得锁并将state=1；如果state=1，则说明有线程目前正在使用共享变量，其他线程必须加入同步队列进行等待。
B：AQS内部是通过Node实体类来表示一个双向链表结构的同步队列，完成线程获取锁的排队工作，当有线程获取锁失败后，就被添加到队列末尾。
Node类是对要访问同步代码的线程的封装，包含了线程本身及其状态waitStatus（它有五种不同的取值，分别表示是否被阻塞、是否等待唤醒、是否已经被取消等），每个Node结点关联其prev结点和next结点（指针），方便线程释放锁后快速唤醒下一个在等待的线程，是一个FIFO的过程；
Node类有两个常量，SHARED和EXCLUSIVE，分别代表共享模式和独占模式，所谓共享模式是一个锁允许多条线程同时操作（信号量Semaphore就是基于AQS的共享模式实现的），独占模式指的是同一个时间段只能有一个线程对共享资源进行操作，多余的请求线程需要排队等待（如ReentranLock）；
C：AQS通过内部类Condition Object构建等待队列（可有多个），当Condition调用wait()方法后，线程将会加入等待队列中，而当Condition调用signal()方法后，线程将从等待队列转移到同步队列中竞争锁。
D：AQS和Condition各自维护了不同的队列，在使用Lock和Condition的时候，其实就是两个队列的互相移动。




13.面试官：请对比下Synchronized 和 ReentrantLock的异同？
（2）回答：ReentrantLock是Lock的实现类，是一个互斥的同步锁；
A：从功能角度上看，ReentrantLock比Synchronized的同步操作更精细（因为可以像普通对象一样使用），甚至实现了Synchronized没有的高级功能，如：
等待可中断：当持有锁的线程长期不释放锁的时候，正在等待的线程可以选择放弃等待，对处理执行时间非常长的同步块很有用。
带超时的获取锁尝试：在指定的时间范围内获取锁，如果时间到了仍然无法获取则返回，可以判断是否有线程在排队等待获取锁。
可以响应中断请求：与Synchronized不同，当获取到锁的线程被中断时，能够响应中断，中断异常将会被抛出，同时锁会被释放。
可以实现公平锁：从锁的释放角度上看，Synchronized在JVM层面上实现的，不但可以通过一些监控工具监控Synchronized的锁定，而且在代码执行出现异常时，JVM会自动释放锁定；但是使用Lock则不行，Lock是通过代码实现的，要保证锁定一定会被释放，就必须将unLock()放到finally{}中。
B：从性能角度上看，Synchronized早期实现比较低效，对比ReentrantLock，大多数场景性能都相差较大，但是在Java6中对其进行了非常多的改进，在竞争不激烈时，Synchronized的性能要优于ReetrantLock；在高竞争情况下，Synchronized的性能会下降几十倍，但是ReetrantLock的性能能维持常态。




14.面试官：上面提到ReentrantLock也是一种可重入锁，那它的底层又是如何实现的？
（2）回答：ReentrantLock内部自定义了同步器Sync（Sync既实现了AQS，又实现了AOS，而AOS提供了一种持有互斥锁的方式），其实就是加锁的时候通过CAS算法，将线程对象放到一个双向链表中，每次获取锁的时候，看下当前维护的那个线程ID和当前请求的线程ID是否一样，一样就可重入了。



15.面试官：除了Synchronized 和 ReentrantLock，你还接触过JUC下中的哪些并发工具？
（2）回答：通常所说的并发包JUC其实就是java.util.concurrent包及其子包下集合了Java并发的各种基础工具类，具体主要包括几个方面：
A：提供了CountDownLatch、CyclicBarrier、Semaphore等，比Synchronized更加高级，可以实现更加丰富多线程操作的同步结构;
B：提供了ConcurrentHashMap、有序的ConcunrrentSkipListMap，或者通过类似快照机制实现线程安全的动态数组CopyOnWriteArrayList等，各种线程安全的容器;
C：提供了ArrayBlockingQueue、SynchorousQueue或针对特定场景的PriorityBlockingQueue等，各种并发队列的实现；

16.面试官：简单说一说ReadWriteLock 和StampedLock 吧？
（2）回答：虽然ReentrantLock和Synchronized简单实用，但是行为上有一定的局限性，要么不占，要么独占；在实际应用场景中，有时候不需要大量竞争的写操作，而是以并发读为主，为了进一步优化并发操作的粒度，Java提供了读写锁；
读写锁基于的原理是：多个读操作不需要互斥，如果读锁试图锁定时，写锁却被某个线程持有时，读锁将无法获得，而只好等待对方操作结束，这样就可以自动保证不会读取到脏数据；
ReadWriteLock代表了一对锁，它在数据量大 且 并发读多、写少的时候，能够比纯同步版本凸显出优势；
读写锁看起来比Synchronized的粒度似乎细一些，但在实际应用中，其表现也并不尽人意，主要还是因为相对比较大的开销；
所以，JDK在后期引入了StampedLock，在提供类似读写锁的同时，还支持优化读模式，优化读是基于这样的假设：大多数情况下读操作并不会和写操作冲突，其逻辑是先试着修改，然后通过validate方法确认是否进入了写模式，如果没有进入，就成功避免了开销；如果进入，则尝试获取读锁。

对ThreadLocal了解多少？

```
对ThreadLocal内存泄露的场景？

实际上 ThreadLocalMap 中使用的 key 为ThreadLocal 的弱引用，而 value 是强引用。弱引用的特点是，如果这个对象持有弱引用，那么在下一次垃圾回收的时候必然会被清理掉。
所以如果 Threadlocal 没有被外部强引用的情况下，在垃圾回收的时候会被清理掉的，这样一来 ThreadLocalMap中使用这个 ThreadLocal 的 key 也会被清理掉。但是value 是强引用，不会被清理，这样一来就会出现 key 为 null的value。假如我们不做任何措施的话，value 永远无法被GC 回收，如果线程长时间不被销毁，可能会产生内存泄露。
因此在使用完threadLocal后，最好手动调用remove方法

内存泄漏和内存溢出的区别是什么？

造成内存泄露典型原因：对象已经死了，无法通过垃圾收集器进行自动回收

堆内存不足：程序创建了过多的对象，超出了 JVM 堆内存的限制

### 逃逸分析

```
对象一定分配在堆内存中吗？什么是逃逸分析技术？

不一定；随着JIT编译期的发展与逃逸分析技术逐渐成熟，所有的对象都分配到堆上也渐渐变得不那么“绝对”了。其实，在编译期间，JIT会对代码做很多优化。其中有一部分优化的目的就是减少内存堆分配压力，其中一种重要的技术叫做逃逸分析。

逃逸分析是指分析指针动态范围的方法，它同编译器优化原理的指针分析和外形分析相关联。当变量(或者对象)在方法中分配后，其指针有可能被返回或者被全局引用，这样就会被其他方法或者线程所引用，这种现象称作指针(或者引用)的逃逸(Escape)
通俗点讲，当一个对象被new出来之后，它可能被外部所调用，如果是作为参数传递到外部了，就称之为方法逃逸。

public static void resStr(){
	User user = new User();
	user.setId(1);
	
}
public static String resStr(){
	User user = new User();
	user.setId(1);
	user.setName("sirl");
	return user.toString();
}
以上两个方法没有发生方法逃逸
public static User resStr(){
	User user = new User();
	user.setId(1);
	user.setName("sirl");
	return user;
}
上边的局部对象user可能会被外部调用，发生方法逃逸

如果对象还有可能被外部线程访问，例如赋值给其他线程中访问的实例遍历，就称为线程逃逸
逃逸强度为：不逃逸<方法逃逸<线程逃逸

逃逸分析的好处
栈上分配
如果确定一个对象不会逃逸到线程之外，那么久可以考虑将这个对象在栈上分配，对象占用的内存随着栈帧出栈而销毁，这样一来，垃圾收集的压力就降低很多。

同步消除
线程同步本身是一个相对耗时的过程，如果逃逸分析能够确定一个变量不会逃逸出线程，无法被其他线程访问，那么这个变量的读写肯定就不会有竞争， 对这个变量实施的同步措施也就可以安全地消除掉。

标量替换
如果一个数据是基本数据类型，不可拆分，它就被称之为标量。把一个Java对象拆散，将其用到的成员变量恢复为原始类型来访问，这个过程就称为标量替换。假如逃逸分析能够证明一个对象不会被方法外部访问，并且这个对象可以被拆散，那么可以不创建对象，直接用创建若干个成员变量代替，可以让对象的成员变量在栈上分配和读写。
```

```
为什么需要Java内存模型？特性是啥？

屏蔽各种硬件和操作系统的内存访问差异
JMM是Java内存模型，也就是Java Memory Model，简称JMM，本身是一种抽象的概念，实际上并不存在，它描述的是一组规则或规范，通过这组规范定义了程序中各个变量（包括实例字段，静态字段和构成数组对象的元素）的访问方式。

可见性（当一个线程修改了共享变量的值时，其他线程能够立即得知这个修改）
原子性（一个操作或一系列操作是不可分割的，要么同时成功，要么同时失败）
有序性（变量赋值操作的顺序与程序代码中的执行顺序一致）
关于有序性：如果在本线程内观察，所有的操作都是有序的；如果在一个线程中观察另一个线程，所有的操作都是无序的。前半句是指“线程内似表现为串行的语义”（Within-Thread As-If-Serial Semantics），后半句是指“指令重排序”现象和“工作内存与主内存同步延迟”现象。

```
并发条件下，变量的同步操作与同步规则有哪些？

同步操作：
lock（锁定）：作用于主内存的变量，把一个变量标识为一条线程独占状态
unlock（解锁）：作用于主内存的变量，把一个处于锁定状态的变量释放出来，释放后的变量才可以被其他线程锁定。
read（读取）：作用于主内存的变量，把一个变量的值从主内存传输到线程的工作内存中，以便以后的load动作使用。
load（载入）：作用于工作内存的变量，它把read操作从主内存中得到的变量值放入工作内存的变量副本中。
use（使用）：作用于工作内存的变量，把工作内存中的一个变量值传递给执行引擎。
assign（赋值）：作用于工作内存的变量，它把一个从执行引擎接收到的值赋值给工作内存的变量。
store（存储）：作用于工作内存的变量，把工作内存中的一个变量的值传送到主内存中，以便以后的write操作。
write（写入）：作用于主内存的变量，它把store操作从工作内存中一个变量的值传送到主内存的变量中。

同步规则：
1：如果要把一个变量从主内存中复制到工作内存，就需要按顺序地执行read和load操作，如果把变量从工作内存中同步回主内存中，就要按顺序地执行store和write操作。但java内存模型只要求上述操作必须按顺序执行，而没有保证必须是连续执行。
2：不允许read和load、store和write操作之一单独出现
3：不允许一个线程丢弃它的最近assign的操作，即变量在工作内存中改变了之后必须同步到主内存中
4：不允许一个线程无原因地（没有发生过任何assign操作）把数据从工作内存同步回主内存中。
5：一个新的变量只能在主内存中诞生，不允许在工作内存中直接使用一个未被初始化（load或assign）的变量。即就是对一个变量实施use和store操作之前，必须先执行过了assign和load操作。
6：一个变量在同一时刻只允许一条线程对其进行lock操作，但lock操作可以被同一条线程重复执行多次，多次执行lock后，只有执行相同次数的unlock操作，变量才会被解锁。lock和unlock必须成对出现。
7：如果一个变量执行lock操作，将会清空工作内存中此变量的值，在执行引擎使用这个变量前需要重新执行load或assign操作初始化变量的值
8：如果一个变量事先没有被lock操作锁定，则不允许对它执行unlock操作；也不允许去unlock一个被其他线程锁定的变量。
9：对一个变量执行unlock操作之前，必须先把此变量同步到主内存中（执行store和write操作）。
```

### Map

```
23.Java 中的 HashMap 的工作原理是什么？

Java 中的 HashMap 是以键值对(key-value)的形式存储元素的。
HashMap 的一些重要的特性是它的容量(capacity)，负载因子(load factor)和扩容极限(threshold resizing)。
HashMap主要由数组和链表组成，他不是线程安全的。核心的点就是put插入数据的过程，get查询数据以及扩容的方式。JDK1.7和1.8的主要区别在于头插和尾插方式的修改，头插容易导致HashMap链表死循环，并且1.8之后加入红黑树对性能有提升。

put插入数据流程
往map插入元素的时候首先通过对key hash然后与数组长度-1进行与运算((n-1)&hash)，都是2的次幂所以等同于取模，但是位运算的效率更高。找到数组中的位置之后，如果数组中没有元素直接存入，反之则判断key是否相同，key相同就覆盖，否则就会插入到链表的尾部，如果链表的长度超过8，则会转换成红黑树，最后判断数组长度是否超过默认的长度*负载因子也就是12，超过则进行扩容。

get查询数据
查询数据相对来说就比较简单了，首先计算出hash值，然后去数组查询，是红黑树就去红黑树查，链表就遍历链表查询就可以了。

resize扩容过程

HashMap 的长度为什么是2的幂次方?

为了能让 HashMap 存取高效，尽量较少碰撞，也就是要尽量把数据分配均匀。
数组下标的计算方法是“ (n - 1) & hash”。（n代表数组长度）。
取余(%)操作中如果除数是2的幂次则等价于与其除数减一的与(&)操作
也就是说 hash%length==hash&(length-1)的前提是 length 是2的 n 次方；
采用二进制位操作 &，相对于%能够提高运算效率，这就解释了 HashMap 的长度为什么是2的幂次方。

```
JAVA8 的 ConcurrentHashMap 为什么放弃了分段锁？还有可优化的地方吗？

分段锁的实现较为复杂，管理多个段的锁需要更多的代码和逻辑，增加了维护的难度
在分段锁的实现中，每个段有一个独立的锁，这意味着在高并发情况下，多个线程可能会争夺同一个段的锁，导致性能瓶颈。
Java 8 的 ConcurrentHashMap 引入了更细粒度的锁机制，使用了“锁分离”技术。具体来说，它使用了节点级别的锁（即使用 synchronized 关键字），允许多个线程同时访问不同的桶（bucket），从而提高了并发性能。
Java 8 中的实现还引入了 CAS（Compare-And-Swap）操作，这种乐观锁的机制使得在插入和更新元素时可以减少锁的使用，从而进一步提高性能

优化的点有
继续使用节点级别的锁，以支持更高的并发性。可以采用 ReentrantLock 来替代 synchronized，以便提供更灵活的锁机制，比如尝试锁、定时锁等

充分利用原子变量和 CAS 操作来处理插入和更新操作，减少锁的持有时间，进一步提升性能。

设计一个动态扩展机制，能够在负载过高时自动扩展容量，避免在高并发情况下的性能下降。

探索无锁数据结构（如使用链表和树的组合）来处理哈希冲突，进一步提高并发性能

考虑内存使用和垃圾回收的影响，设计时应考虑如何减少内存碎片和优化内存使用

可以考虑引入读写分离机制，比如使用读写锁（ReadWriteLock），提高读操作的并发性，同时在写操作时保证数据一致性。

```java
import java.util.concurrent.atomic.AtomicInteger;
import java.util.concurrent.atomic.AtomicReference;
import java.util.concurrent.locks.ReentrantReadWriteLock;

class Node<K, V> {
    final K key;
    volatile V value;
    final AtomicReference<Node<K, V>> next; // 链表中的下一个节点
    final ReentrantReadWriteLock lock; // 读写锁

    Node(K key, V value) {
        this.key = key;
        this.value = value;
        this.next = new AtomicReference<>(null);
        this.lock = new ReentrantReadWriteLock();
    }
}

public class SimpleConcurrentHashMap<K, V> {
    private volatile Node<K, V>[] table; // 哈希表
    private final int initialCapacity; // 初始容量
    private final float loadFactor; // 负载因子
    private volatile int size; // 当前大小
    private final AtomicInteger threshold; // 扩展阈值

    @SuppressWarnings("unchecked")
    public SimpleConcurrentHashMap(int initialCapacity, float loadFactor) {
        this.initialCapacity = initialCapacity;
        this.loadFactor = loadFactor;
        this.size = 0;
        this.threshold = new AtomicInteger((int) (initialCapacity * loadFactor));
        this.table = new Node[initialCapacity];
    }

    private int hash(K key) {
        return (key.hashCode() & 0x7FFFFFFF) % table.length;
    }

    public V put(K key, V value) {
        if (size >= threshold.get()) {
            resize(); // 检查并扩展
        }

        int index = hash(key);
        Node<K, V> newNode = new Node<>(key, value);

        Node<K, V> current = table[index];
        ReentrantReadWriteLock lock = new ReentrantReadWriteLock();
        lock.writeLock().lock();
        try {
            if (current == null) {
                table[index] = newNode; // 桶为空，直接插入
                size++;
                return null;
            }

            // 遍历链表寻找插入位置
            while (true) {
                current.lock.readLock().lock();
                try {
                    if (current.key.equals(key)) {
                        V oldValue = current.value;
                        current.value = value; // 更新值
                        return oldValue;
                    }
                    Node<K, V> next = current.next.get();
                    if (next == null) {
                        current.next.set(newNode); // 在链表末尾插入
                        size++;
                        return null;
                    }
                    current.lock.readLock().unlock();
                    current = next;
                } finally {
                    current.lock.readLock().unlock();
                }
            }
        } finally {
            lock.writeLock().unlock();
        }
    }

    public V get(K key) {
        int index = hash(key);
        Node<K, V> current = table[index];

        while (current != null) {
            current.lock.readLock().lock();
            try {
                if (current.key.equals(key)) {
                    return current.value; // 找到返回值
                }
                current = current.next.get();
            } finally {
                current.lock.readLock().unlock();
            }
        }
        return null; // 未找到
    }

    public V remove(K key) {
        int index = hash(key);
        ReentrantReadWriteLock lock = new ReentrantReadWriteLock();
        lock.writeLock().lock();
        try {
            Node<K, V> current = table[index];
            Node<K, V> previous = null;

            while (current != null) {
                current.lock.readLock().lock();
                try {
                    if (current.key.equals(key)) {
                        if (previous == null) {
                            table[index] = current.next.get(); // 删除头节点
                        } else {
                            previous.next.set(current.next.get()); // 删除中间节点
                        }
                        size--;
                        return current.value; // 返回被删除的值
                    }
                    previous = current;
                    current = current.next.get();
                } finally {
                    current.lock.readLock().unlock();
                }
            }
            return null; // 未找到
        } finally {
            lock.writeLock().unlock();
        }
    }

    @SuppressWarnings("unchecked")
    private void resize() {
        int newCapacity = table.length * 2;
        Node<K, V>[] newTable = new Node[newCapacity];

        for (Node<K, V> node : table) {
            while (node != null) {
                int newIndex = (node.key.hashCode() & 0x7FFFFFFF) % newCapacity;
                Node<K, V> next = node.next.get();
                node.next.set(newTable[newIndex]); // 迁移节点
                newTable[newIndex] = node; // 更新新表
                node = next;
            }
        }

        table = newTable; // 更新哈希表引用
        threshold.set((int) (newCapacity * loadFactor)); // 更新扩展阈值
    }

    public int size() {
        return size; // 返回当前大小
    }
}

class HashMapTest {
    public static void main(String[] args) {
        HashMap<Integer, User> users = new HashMap<>();
        users.put(1, new User("张三", 25));
        users.put(3,new User("李四",22));
        users.put(2, new User("王五", 28));

         * {2=User [name=王五, age=28], 1=User [name=张三, age=25], 3=User [name=李四, age=22]}
         */
    }

    public static HashMap<Integer, User> sortHashMap(HashMap<Integer, User> map) {
        // 首先拿到 map 的键值对集合
        Set<Map.Entry<Integer, User>> entrySet = map.entrySet();
        // 将 set 集合转为 List 集合，为什么，为了使用工具类的排序方法
        List<Map.Entry<Integer,User>> list = new ArrayList<Map.Entry<Integer, User>>(entrySet);
        // 使用 Collections 集合工具类对 list 进行排序，排序规则使用匿名内部类来实现
        Collections.sort(list, new Comparator<Map.Entry<Integer, User>>() {
            @Override
            public int compare(Map.Entry<Integer, User> o1, Map.Entry<Integer, User> o2) {
                //按照要求根据 User 的 age 的倒序进行排
                return o2.getValue().getAge() - o1.getValue().getAge();
            }
        });
        //创建一个新的有序的 HashMap 子类的集合
        LinkedHashMap<Integer, User> linkedHashMap = new LinkedHashMap<Integer, User>();
        //将 List 中的数据存储在 LinkedHashMap 中
        for (Map.Entry<Integer,User> entry : list) {

        }
        return linkedHashMap;
    }

39.Java 堆的结构是什么样子的？什么是堆中的永久代(Perm Gen space)?

JVM 的堆是运行时数据区，所有类的实例和数组都是在堆上分配内存。它在 JVM 启动的时候被创建。对象所占的堆内存是由自动内存管理系统也就是垃圾收集器回收。 

堆内存是由存活和死亡的对象组成的。存活的对象是应用可以访问的，不会被垃圾回收。死亡的对象是应用不可访问尚且还没有被垃圾收集器回收掉的对象。一直到垃圾收集器把这些对象回收掉之前，他们会一直占据堆内存空间。

java堆从GC角度可分为老年代和新生代。其中新生代又分为Eden区和两个Survivor 区（以下简称S0区和S1区）

因为JAVA对象90%以上的对象都是朝生夕死的，其中GC回收的成本很高，为了提高性能所以将新生成的对象放在Eden区，将扛过多次GC的“老家伙”放在老年代

因为Eden区的绝大部分对象寿命很短，那么Eden每次满了清理垃圾，存活的对象被迁移到老年区，老年区满了，就会触发Full GC，Full GC是非常耗时的，设立s区的一个目的就是在Eden区和老年代中增加一个缓冲池，放一些“年纪不够老”的对象，增加垃圾回收性能

Survivor 区也会进行垃圾回收，但是并非主动进行的垃圾回收，是Eden区在进行垃圾回收的时候顺带回收、默认Eden区和 s0 ,s1 区的比例是 8:1:1。

（复制算法）设置两个Survivor区最大的好处就是解决了碎片化，刚刚新建的对象在Eden中，经历一次Minor GC，Eden中的存活对象就会被移动到S0，Eden被清空；等Eden区再满了，就再触发一次Minor GC，Eden和S0中的存活对象又会被复制送入S1区（这个过程非常重要，因为这种复制算法保证了S1中来自S0和Eden两部分的存活对象占用连续的内存空间，避免了碎片化的发生），接着新对象继续分配在Eden区和另外那块开始被使用的Survivor区，然后始终保持一块Survivor区是空着的，就这样一直循环使用这三块内存区域
```

```
JVM 的垃圾回收算法有哪些？

1：可达性分析算法（标记阶段）
原理： 可达性分析算法是以根对象集合（GCRoots）为起始点，按照从上至下的方式搜索被根对象集合所连接的目标对象是否可达。 虚拟机栈、本地方法栈、方法区、字符串常量池 等地方对堆空间进行引用的，都可以作为GC Roots进行可达性分析。

2：标记-清除算法（年轻代清除阶段）
原理： 当堆中的有效内存空间（available memory）被耗尽的时候，就会停止整个程序（也被称为stop the world），然后进行两项工作，第一项则是标记，第二项则是清除 标记：Collector从引用根节点开始遍历，标记所有被引用的对象。一般是在对象的Header中记录为可达对象。 清除：Collector对堆内存从头到尾进行线性的遍历，如果发现某个对象在其Header中没有标记为可达对象，则将其回收
缺点：
标记清除算法的效率不算高。
在进行GC的时候，需要停止整个应用程序，用户体验较差。
这种方式清理出来的空闲内存是不连续的，产生内碎片，需要维护一个空闲列表。

3：复制算法（年轻代清除阶段）
因为标记-清除算法的缺点，由此发明了复制算法。 原理： 将活着的内存空间分为两块，每次只使用其中一块，在垃圾回收时将正在使用的内存中的存活对象复制到未被使用的内存块中，之后清除正在使用的内存块中的所有对象，交换两个内存的角色，最后完成垃圾回收。
优点： 复制过去以后保证空间的连续性，不会出现“碎片”问题。
缺点： 需要多余的内存空间。

4：标记整理算法（老年代清除阶段）
背景： 复制算法的高效性是建立在存活对象少、垃圾对象多的前提下的。这种情况在新生代经常发生，但是在老年代，更常见的情况是大部分对象都是存活对象。如果依然使用复制算法，由于存活对象较多，复制的成本也将很高。因此，基于老年代垃圾回收的特性，需要使用其他的算法。
原理： 第一阶段和标记清除算法一样，从根节点开始标记所有被引用对象。 第二阶段将所有的存活对象压缩到内存的一端，按顺序排放。之后，清理边界外所有的空间。
标记-压缩算法的最终效果等同于标记-清除算法执行完成后，再进行一次内存碎片整理，因此，也可以把它称为标记-清除-压缩（Mark-Sweep-Compact）算法。
优点：
消除了标记-清除算法当中，内存区域分散的缺点，我们需要给新对象分配内存时，JVM只需要持有一个内存的起始地址即可。 消除了复制算法当中，内存减半的高额代价。
缺点： 从效率上来说，标记-整理算法要低于复制算法。 移动对象的同时，如果对象被其他对象引用，则还需要调整引用的地址 移动过程中，需要全程暂停用户应用程序。即：STW。

5：分代收集算法
背景： 不同生命周期的对象可以采取不同的收集方式，以便提高回收效率。一般是把Java堆分为新生代和老年代，这样就可以根据各个年代的特点使用不同的回收算法，以提高垃圾回收的效率。
年轻代：复制算法 老年代：由标记-清除或者是标记-清除与标记-整理的混合实现。

6：增量收集算法
原理： 如果一次性将所有的垃圾进行处理，需要造成系统长时间的停顿，那么就可以让垃圾收集线程和应用程序线程交替执行。每次，垃圾收集线程只收集一小片区域的内存空间，接着切换到应用程序线程。依次反复，直到垃圾收集完成。
缺点： 使用这种方式，由于在垃圾回收过程中，间断性地还执行了应用程序代码，所以能减少系统的停顿时间。但是，因为线程切换和上下文转换的消耗，会使得垃圾回收的总体成本上升，造成系统吞吐量的下降。

7：分区算法（G1 收集器）
原理： 分区算法将整个堆空间划分成连续的不同小区间。 每一个小区间都独立使用，独立回收。这种算法的好处是可以控制一次回收多少个小区间。

8：三色标记算法
用于垃圾回收器升级，将STW变为并发标记，STW就是在标记垃圾的时候，必须暂停程序，而使用并发标记，就是程序一边运行一边标记垃圾；可以避免重复扫描对象，提升标记阶段的效率
三色标记法是根据可达性分析，从根节点开始进行遍历访问，在遍历对象过程中，按照是否检查过的条件将对象标记为三种颜色
白色，该对象没有标记过，是垃圾对象
灰色，该对象已经被标记过，但该对象下的属性没有完全被标记，gc的时候需要从此对象中寻找垃圾
黑色，该对象已经被标记过，且该对象下的属性也完全标记了，是程序运行所需要的对象
三色标记算法是根可达的一种实现方案，目的是找出所有可达对象，但是会存在多标和漏标问题
```

分析下对象可达性状态流转

强可达（Strongly Reachable），就是当一个对象可以有一个或多个线程可以不通过各种引用访问到的情况。比如，我们新创建一个对象，那么创建它的线程对它就是强可达。

软可达（Softly Reachable），就是当我们只能通过软引用才能访问到对象的状态。

弱可达（Weakly Reachable），类似前面提到的，就是无法通过强引用或者软引用访问，只能通过弱引用访问时的状态。这是十分临近 finalize 状态的时机，当弱引用被清除的时候，就符合 finalize 的条件了。

幻象可达（Phantom Reachable），上面流程图已经很直观了，就是没有强、软、弱引用关联，并且 finalize 过了，只有幻象引用指向这个对象的时候。

不可达（unreachable），意味着对象可以被清除了。

所有引用类型，都是抽象类 java.lang.ref.Reference 的子类，它提供了 get() 方法，除了幻象引用（因为 get 永远返回 null），如果对象还没有被销毁，都可以通过 get 方法获取原有对象。这意味着，利用软引用和弱引用，我们可以将访问到的对象，重新指向强引用，也就是人为的改变了对象的可达性状态！所以，对于软引用、弱引用之类，垃圾收集器可能会存在二次确认的问题，以保证处于弱引用状态的对象，没有改变为强引用。如果我们错误的保持了强引用（比如，赋值给了 static 变量），那么对象可能就没有机会变回类似弱引用的可达性状态了，就会产生内存泄漏。

说一下 JVM 的垃圾回收器 CMS和G1？优缺点是什么？

CMS：以获取最短回收停顿时间为目标的收集器，基于并发“标记清理”实现；优点是：并发，低停顿；缺点是：对CPU非常敏感，无法处理浮动垃圾，会产生大量空间碎片
垃圾收集分为四步
初始标记，单线程运行，需要停止程序，标记根可达对象
并发标记，无需停止程序，和用户线程同时运行，从根可达对象开始遍历整个对象图
重新标记，多线程运行，需要停止程序，标记并发标记阶段产生对象
并发清除，无需停止程序，和用户线程同时运行，清理掉标记的死亡对象


G1：是一款面向服务端应用的垃圾收集器；实现了更精细的控制，可预测的停顿时间，内存碎片的控制，优先级处理等；

每个区域都通过指针碰撞来分配空间，这样就避免了收集整个堆，而是按照区域进行收集，同时维护一个优先级列表，跟踪各个区域回收的价值，优先收集价值高的区域
垃圾收集分为四步
初始标记，需要停止程序STW，标记了从根可达对象
并发标记，和用户线程并发执行，从根可达对象开始对堆中的对象进行可达性分析，递归扫描，找出回收对象
最终标记，STW，标记并发标记中产生的垃圾
筛选回收，制定回收计划，选择多个区域构成回收集，把回收集中存活对象复制到空的区域中，再清理掉旧的区域全部空间，需要STW

Exception 和 Error 都是 Throwable 的子类。
Exception 是程序正常运行中，可以预料的意外情况，可能并且应该被捕获，进行相应处理。
Exception 又分为可检查（checked）异常和不检查（unchecked）异常，可检查异常在源代码里必须显式地进行捕获处理，这是编译期检查的一部分。不检查异常就是所谓的运行时异常，类似 NullPointerException、ArrayIndexOutOfBoundsException 之类，通常是可以编码避免的逻辑错误，具体根据需要来判断是否需要捕获，并不会在编译期强制要求。
Error 是指在正常情况下，不大可能出现的情况，绝大部分的 Error 都会导致程序（比如 JVM 自身）处于非正常的、不可恢复状态。既然是非正常情况，所以不便于也不需要捕获，常见的比如 OutOfMemoryError 之类，都是 Error 的子类。
```

```
ClassNotFoundException和NoClassDefFoundError的区别和产生原因是什么？

ClassNotFoundException的产生原因：Java支持使用Class.forName方法来动态地加载类，任意一个类的类名如果被作为参数传递给这个方法都将导致该类被加载到JVM内存中，如果这个类在类路径中没有被找到，那么此时就会在运行时抛出ClassNotFoundException异常。

NoClassDefFoundError产生的原因在于：如果JVM或者ClassLoader实例尝试加载（可以通过正常的方法调用，也可能是使用new来创建新的对象）类的时候却找不到类的定义。要查找的类在编译的时候是存在的，运行的时候却找不到了。这个时候就会导致NoClassDefFoundError.造成该问题的原因可能是打包过程漏掉了部分类，或者jar包出现损坏或者篡改。解决这个问题的办法是查找那些在开发期间存在于类路径下但在运行期间却不在类路径下的类。
```

异常处理的原则有哪些？

第一，尽量不要捕获类似 Exception 这样的通用异常，而是应该捕获特定异常。
第二，不要生吞（swallow）异常。否则会加大排查问题的难度
第三，生产环境中不要打印堆栈信息，而是要把异常记录到日志中
第四，对于有可能出现NPE的地方，要提早做处理，否则不容易定位空指针的位置
Objects. requireNonNull(filename);
第五，函数返回值有两种类型：值类型与对象引用。对于对象引用，要特别小心，如果在finally代码块中对函数返回的对象成员属性进行了修改，即使不在finally块中显式调用return语句，这个修改也会作用于返回值上。
第六，继承某个异常时，重写方法时，要么不抛出异常，要么抛出一模一样的异常
第七，当一个try后跟了很多个catch时，必须先捕获小的异常再捕获大的异常。

为什么说Java 语言的 Checked Exception 也许是个设计错误？

Checked Exception 的假设是我们捕获了异常，然后恢复程序。但是，其实我们大多数情况下，根本就不可能恢复。Checked Exception 的使用，已经大大偏离了最初的设计目的。
Checked Exception 不兼容 functional 编程，如果你写过 Lambda/Stream 代码，相信深有体会。

Java的异常处理机制可能存在什么性能问题？

try-catch 代码段会产生额外的性能开销，或者换个角度说，它往往会影响 JVM 对代码进行优化，所以建议仅捕获有必要的代码段，尽量不要一个大的 try 包住整段的代码；与此同时，利用异常控制代码流程，也不是一个好主意，远比我们通常意义上的条件语句（if/else、switch）要低效。

Java 每实例化一个 Exception，都会对当时的栈进行快照，这是一个相对比较重的操作。如果发生的非常频繁，这个开销可就不能被忽略了。

当我们的服务出现反应变慢、吞吐量下降的时候，检查发生最频繁的 Exception 也是一种思路。

47.finally 代码块和 finalize()方法有什么区别？有什么机制可以替换 finalize 吗？

无论是否抛出异常，finally 代码块都会执行，它主要是用来释放应用占用的资源。
finalize()方法是 Object 类的一个 protected 方法，它是在对象被垃圾回收之前由 Java 虚拟机来调用的。它的设计目的是保证对象在被垃圾收集前完成特定资源的回收。如果没有特别的原因，不要实现 finalize 方法，也不要指望利用它来进行资源回收。你无法保证 finalize 什么时候执行，执行的是否符合预期。使用不当会影响性能，导致程序死锁、挂起等。finalize 的执行是和垃圾收集关联在一起的，一旦实现了非空的 finalize 方法，就会导致相应对象回收呈现数量级上的变慢

Java 平台目前在逐步使用 java.lang.ref.Cleaner 来替换掉原有的 finalize 实现。Cleaner 的实现利用了幻象引用（PhantomReference），这是一种常见的所谓 post-mortem 清理机制。利用幻象引用和引用队列，我们可以保证对象被彻底销毁前做一些类似资源回收的工作，比如关闭文件描述符（操作系统有限的资源），它比 finalize 更加轻量、更加可靠。
吸取了 finalize 里的教训，每个 Cleaner 的操作都是独立的，它有自己的运行线程，所以可以避免意外死锁等问题。

## RMI

```
78.什么是 RMI？

Java 远程方法调用(Java RMI)是 Java API 对远程过程调用(RPC)提供的面向对象的等价形式，支持直接传输序列化的 Java 对象和分布式垃圾回收。远程方法调用可以看做是激活远程正在运行的对象上的方法的步骤。RMI 对调用者是位置透明的，因为调用者感觉方法是执行在本地运行的对象上的
```

```
79.RMI 体系结构的基本原则是什么？

RMI 体系结构是基于一个非常重要的行为定义和行为实现相分离的原则。RMI 允许定义行为的代码和实现行为的代码相分离，并且运行在不同的 JVM 上。 
```

```
80.RMI 体系结构分哪几层？

存根和骨架层(Stub and Skeleton layer)：这一层对程序员是透明的，它主要负责拦截客户端发出的方法调用请求，然后把请求重定向给远程的 RMI 服务。 
远程引用层(Remote Reference Layer)：RMI 体系结构的第二层用来解析客户端对服务端远程对象的引用。这一层解析并管理客户端对服务端远程对象的引用。连接是点到点的。 
传输层(Transport layer)：这一层负责连接参与服务的两个 JVM。这一层是建立在网络上机器间的 TCP/IP 连接之上的。它提供了基本的连接服务，还有一些防火墙穿透策略。
```

```
81.RMI 中的远程接口(Remote Interface)扮演了什么样的角色？

远程接口用来标识哪些方法是可以被非本地虚拟机调用的接口。远程对象必须要直接或者是间接实现远程接口。实现了远程接口的类应该声明被实现的远程接口，给每一个远程对象定义构造函数，给所有远程接口的方法提供实现。
```

```
82.java.rmi.Naming 类扮演了什么样的角色？

java.rmi.Naming 类用来存储和获取在远程对象注册表里面的远程对象的引用。Naming 类的每一个方法接收一个 URL 格式的 String 对象作为它的参数。 
```

```
83.RMI 的绑定(Binding)是什么意思？

绑定是为了查询找远程对象而给远程对象关联或者是注册以后会用到的名称的过程。远程对象可以使用 Naming 类的 bind()或者 rebind()方法跟名称相关联。 
```

```
84.Naming 类的 bind()和 rebind()方法有什么区别？

bind()方法负责把指定名称绑定给远程对象，rebind()方法负责把指定名称重新绑定到一个新的远程对象。如果那个名称已经绑定过了，先前的绑定会被替换掉。
```

```
85.让 RMI 程序能正确运行有哪些步骤？

编译所有的源文件。 
使用 rmic 生成 stub。 
启动 rmiregistry。 
启动 RMI 服务器。 
运行客户端程序。
```

```
86.RMI 的 stub 扮演了什么样的角色？

远程对象的 stub 扮演了远程对象的代表或者代理的角色。调用者在本地 stub 上调用方法，它负责在远程对象上执行方法。当 stub 的方法被调用的时候，会经历以下几个步骤： 
初始化到包含了远程对象的 JVM 的连接。 
序列化参数到远程的 JVM。 
等待方法调用和执行的结果。 
反序列化返回的值或者是方法没有执行成功情况下的异常。 
把值返回给调用者。
```

```
87.什么是分布式垃圾回收(DGC)？它是如何工作的？

DGC 叫做分布式垃圾回收。RMI 使用 DGC 来做自动垃圾回收。因为 RMI 包含了跨虚拟机的远程对象的引用，垃圾回收是很困难的。DGC 使用引用计数算法来给远程对象提供自动内存管理。
```

90.解释下 Serialization 和 Deserialization。

Java 提供了一种叫做对象序列化的机制，他把对象表示成一连串的字节，里面包含了对象的数据，对象的类型信息，对象内部的数据的类型信息等等。因此，序列化可以看成是为了把对象存储在磁盘上或者是从磁盘上读出来并重建对象而把对象扁平化的一种方式。反序列化是把对象从扁平状态转化成活动对象的相反的步骤。

```
什么是二叉树，二叉搜索树，平衡二叉树，红黑树？从结构，特点，优点，缺点，适用场景是什么？

二叉树 (Binary Tree)
结构
每个节点最多有两个子节点（左、右子树），无顺序约束。
特点
形态多样（如满二叉树、完全二叉树），无自动平衡机制。
优点
结构简单，实现容易；适合表达层次关系或递归操作。
缺点
无序导致查找效率低（最坏时间复杂度 O(n)）；易退化成链表。
适用场景
构建其他高级树结构的基础，或简单层次化数据存储（如表达式树、文件系统目录结构）。


二叉搜索树 (BST, Binary Search Tree)
结构
在二叉树基础上，满足：左子树所有节点值 < 根节点值 < 右子树所有节点值。
特点
中序遍历结果为有序序列；查找效率依赖树的平衡性。
优点
插入、删除、查找的平均时间复杂度为 O(log n)（平衡时）。
缺点
若插入数据有序，树会退化为链表，最坏时间复杂度 O(n)。
适用场景
数据随机插入且无需频繁修改的场景（需配合平衡机制优化）。


平衡二叉树 (AVL 树)
结构
在 BST 基础上，要求任意节点的左右子树高度差 ≤ 1。
特点
严格平衡，通过旋转（左旋/右旋）调整树结构。
优点
查找效率极高，严格保证 O(log n) 时间复杂度。
缺点
频繁插入/删除时需多次旋转调整，维护平衡开销大。
适用场景
读多写少的场景（如数据库索引、文件系统元数据管理）。


红黑树 (Red-Black Tree)
结构
在 BST 基础上，通过颜色标记和规则约束平衡性：
节点为红或黑；
根和叶子（NIL）为黑；
红节点的子节点必为黑；
任意路径的黑节点数相同。

特点
近似平衡（最长路径 ≤ 2 倍最短路径），插入/删除效率高。
优点
插入/删除操作平均仅需 O(1) 次旋转，综合性能优秀（时间复杂度 O(log n)）。
缺点
查找效率略低于 AVL 树（树相对更高）。
适用场景
高频读写场景（如 Java TreeMap、C++ STL map、Linux 进程调度）。

字节流和字符流的区别？

字节流读取的时候，读到一个字节就返回一个字节；

字符流使用了字节流读到一个或多个字节（中文对应的字节数是两个，在 UTF-8 码表中是 3 个字节）时。先去查指定的编码表，将查到的字符返回。这个过程还算是非常耗时，并且，如果我们不知道编码类型就很容易出现乱码问题。所以， I/O 流就干脆提供了一个直接操作字符的接口，方便我们平时对字符进行流操作。

字节流可以处理所有类型数据，如：图片，MP3，AVI视频文件，而字符流只能处理字符数据。
只要是处理纯文本数据，就要优先考虑使用字符流，除此之外都用字节流。

字节流主要是操作 byte 类型数据，以 byte 数组为准，主要操作类就是 OutputStream、InputStream;
字符流处理的单元为 2 个字节的 Unicode 字符，分别操作字符、字符数组或字符串;
字节流处理单元为 1 个字节，操作字节和字节数组。

如何实现对象克隆？

1:实现 Cloneable 接口并重写 Object 类中的 clone()方法；
2:实现 Serializable 接口，通过对象的序列化和反序列化实现克隆，可以实现真正的深度克隆
注意：基于序列化和反序列化实现的克隆不仅仅是深度克隆，更重要的是通过泛型限定，可以检查出要克隆的对象是否支持序列化，这项检查是编译器完成的，不是在运行时抛出异常，这种是方案明显优于使用 Object 类的 clone 方法克隆对象。让问题在编译的时候暴露出来总是好过把问题留到运行时。
浅克隆：创建一个新对象，新对象的属性和原来对象完全相同，对于非基本类型属性，仍指向原有属性所指向的对象的内存地址。
深克隆：创建一个新对象，属性中引用的其他对象也会被克隆，不再指向原有对象地址。

```
BIO,NIO,AIO 有什么区别?

BIO (Blocking I/O): 同步阻塞I/O模式，数据的读取写入必须阻塞在一个线程内等待其完成。可以让每一个连接专注于自己的 I/O 并且编程模型简单，也不用过多考虑系统的过载、限流等问题。

NIO (New I/O): NIO是一种同步非阻塞的I/O模型，在Java 1.4 中引入了NIO框架，对应 java.nio 包，提供了 Channel , Selector，Buffer等抽象。NIO中的N可以理解为Non-blocking，不单纯是New。它支持面向缓冲的，基于通道的I/O操作方法。 NIO提供了与传统BIO模型中的 Socket 和 ServerSocket 相对应的 SocketChannel 和 ServerSocketChannel 两种不同的套接字通道实现,两种通道都支持阻塞和非阻塞两种模式。阻塞模式使用就像传统中的支持一样，比较简单，但是性能和可靠性都不好；非阻塞模式正好与之相反。对于低负载、低并发的应用程序，可以使用同步阻塞I/O来提升开发速率和更好的维护性；对于高负载、高并发的（网络）应用，应使用 NIO 的非阻塞模式来开发

AIO (Asynchronous I/O):它是异步非阻塞的IO模型。异步 IO 是基于事件和回调机制实现的，也就是应用操作之后会直接返回，不会堵塞在那里，当后台处理完成，操作系统会通知相应的线程进行后续的操作。
```

常见的http数据压缩算法有哪些，压缩算法是什么？

GZIP压缩
广泛用于 Web 传输中。它通过减少数据的大小来提高传输效率
Gzip 是基于 DEFLATE 算法的压缩工具，主要使用两种技术
LZ77 算法：这是一种字典压缩技术，通过查找重复的字符串来减少数据量。LZ77 使用滑动窗口技术，记录之前出现的字符串，并用指针和长度来替代重复字符串
Huffman 编码：这是一种变长编码技术，将出现频率高的字符用较短的编码表示，频率低的字符用较长的编码表示，从而进一步减少数据大小。

LZ77 压缩：同样使用滑动窗口来查找重复字符串
Huffman 编码：对 LZ77 输出的数据进行 Huffman 编码，但 Deflate 允许使用动态生成的 Huffman 树，从而实现更好的压缩效果。

Brotli压缩
一种相对较新的压缩算法，旨在提供更好的压缩比率。它被广泛用于现代 Web 浏览器和服务器
字典压缩：Brotli 使用静态和动态字典，静态字典包含常见的词和短语，动态字典则根据输入数据生成。
LZ77 算法：采用类似于 LZ77 的技术查找重复字符串。
Huffman 编码：使用变长编码的 Huffman 方法，但 Brotli 允许使用更复杂的编码方案，基于输入数据的统计特性。
上下文建模：Brotli 还使用上下文建模技术，考虑前面的字符，以提高编码的效率。

解释一下滑动窗口的实现原理，并写一个小demo

滑动窗口是一种用于数据压缩（如 LZ77 算法）的技术，它通过维护一个固定大小的窗口来查找和替换重复的字符串。窗口分为两部分：
搜索缓冲区（或称为字典）：存储之前已经处理过的数据，通常在窗口的左侧。
前瞻缓冲区：当前正在处理的数据，通常在窗口的右侧。

步骤为
1：初始化窗口：设定窗口的大小，通常为固定的字节数。
2：查找匹配：在搜索缓冲区中查找前瞻缓冲区的内容，寻找最长的匹配字符串。
3：生成指令：如果找到匹配，生成一个指令，通常包含 偏移量（匹配字符串在搜索缓冲区中的起始位置）和长度（匹配字符串的长度）；如果没有找到匹配，直接输出当前字符
4：更新窗口：每次处理完一个字符后，窗口向右滑动，搜索缓冲区向前移动，前瞻缓冲区扩展，继续处理下一个字符。
5：重复2-4的步骤，直到处理完所有数据

import java.util.ArrayList;
import java.util.List;

public class LZ77Compression {
    public static void main(String[] args) {
        String input = "ABABABABA";
        List<Triple> compressed = lz77Compress(input, 4);
        
        // 输出压缩结果
        for (Triple t : compressed) {
            System.out.println(t);
        }
    }

    static class Triple {
        int offset;   // 偏移量
        int length;   // 长度
        char next;    // 下一个字符

        Triple(int offset, int length, char next) {
            this.offset = offset;
            this.length = length;
            this.next = next;
        }

        @Override
        public String toString() {
            return "(" + offset + ", " + length + ", '" + next + "')";
        }
    }

    public static List<Triple> lz77Compress(String input, int windowSize) {
        List<Triple> result = new ArrayList<>();
        int i = 0;

        while (i < input.length()) {
            int matchLength = 0;
            int matchOffset = 0;

            // 确定搜索缓冲区的起始位置
            int searchStart = Math.max(0, i - windowSize);
            String searchBuffer = input.substring(searchStart, i);

            // 查找最长匹配
            for (int j = 0; j < searchBuffer.length(); j++) {
                int length = 0;
                while (i + length < input.length() && 
                       j + length < searchBuffer.length() && 
                       searchBuffer.charAt(j + length) == input.charAt(i + length)) {
                    length++;
                }
                if (length > matchLength) {
                    matchLength = length;
                    matchOffset = searchBuffer.length() - j; // 计算偏移量
                }
            }

            // 如果没有匹配，输出当前字符
            char nextChar = (matchLength > 0) ? input.charAt(i + matchLength) : input.charAt(i);
            result.add(new Triple(matchOffset, matchLength, nextChar));

            // 更新当前索引
            i += matchLength + 1; // 移动到下一个字符
        }

        return result;
    }
}

第二个字符 'B' 偏移量为 1，长度为 1（匹配 'A'）。

TCP/IP 四层体系结构是怎样的？

应用层
应用层( application-layer ）的任务是通过应用进程间的交互来完成特定网络应 用。应用层协议定义的是应用进程（进程：主机中正在运行的程序）间的通信和 交互的规则。对于不同的网络应用需要不同的应用层协议。在互联网中应用层协议很多，如域 名系统 DNS，支持万维网应用的 HTTP 协议，支持电子邮件的 SMTP 协议等 等。

运输层
运输层(transport layer)的主要任务就是负责向两台主机进程之间的通信提供通 用的数据传输服务。应用进程利用该服务传送应用层报文。 运输层主要使用一下两种协议
1. 传输控制协议-TCP：提供面向连接的，可靠的数据传输服务。
2. 用户数据协议-UDP：提供无连接的，尽大努力的数据传输服务（不 保证数据传输的可靠性）。

网络层
网络层的任务就是选择合适的网间路由和交换结点，确保计算机通信的数据及时 传送。在发送数据时，网络层把运输层产生的报文段或用户数据报封装成分组和 包进行传送。在TCP/IP 体系结构中，由于网络层使用 IP 协议，因此分组也叫 IP 数据报 ，简称数据报。

数据链路层
数据链路层(data link layer)通常简称为链路层。两台主机之间的数据传输，总 是在一段一段的链路上传送的，这就需要使用专门的链路层的协议。
在两个相邻节点之间传送数据时，数据链路层将网络层交下来的 IP 数据报组装 成帧，在两个相邻节点间的链路上传送帧。每一帧包括数据和必要的控制信息 （如同步信息，地址信息，差错控制等）。
在接收数据时，控制信息使接收端能够知道一个帧从哪个比特开始和到哪个比特结束。

物理层
在物理层上所传送的数据单位是比特。 物理层(physical layer)的作用是实现相 邻计算机节点之间比特流的透明传送，尽可能屏蔽掉具体传输介质和物理设备的 差异。使其上面的数据链路层不必考虑网络的具体传输介质是什么。“透明传送 比特流”表示经实际电路传送后的比特流没有发生变化，对传送的比特流来说， 这个电路好像是看不见的。

```
介绍下三次挥手和四次握手流程？

三次挥手
第一次握手：客户端要向服务端发起连接请求，首先客户端随机生成一个起始序列号ISN(比如是100)，那客户端向服务端发送的报文段包含SYN标志位(也就是SYN=1)，序列号seq=100。
第二次握手：服务端收到客户端发过来的报文后，发现SYN=1，知道这是一个连接请求，于是将客户端的起始序列号100存起来，并且随机生成一个服务端的起始序列号(比如是300)。然后给客户端回复一段报文，回复报文包含SYN和ACK标志(也就是SYN=1,ACK=1)、序列号seq=300、确认号ack=101(客户端发过来的序列号+1)。
第三次握手：客户端收到服务端的回复后发现ACK=1并且ack=101,于是知道服务端已经收到了序列号为100的那段报文；同时发现SYN=1，知道了服务端同意了这次连接，于是就将服务端的序列号300给存下来。然后客户端再回复一段报文给服务端，报文包含ACK标志位(ACK=1)ack=301(服务端序列号+1)、seq=101(第一次握手时发送报文是占据一个序列号的，所以这次seq就从101开始，需要注意的是不携带数据的ACK报文是不占据序列号的，所以后面第一次正式发送数据时seq还是101)。当服务端收到报文后发现ACK=1并且ack=301，就知道客户端收到序列号为300的报文了，就这样客户端和服务端通过TCP建立了连接。

四次挥手
第一次挥手：当客户端的数据都传输完成后，客户端向服务端发出连接释放报文(当然数据没发完时也可以发送连接释放报文并停止发送数据)，释放连接报文包含FIN标志位(FIN=1)、序列号seq=1101(100+1+1000，其中的1是建立连接时占的一个序列号)。需要注意的是客户端发出FIN报文段后只是不能发数据了，但是还可以正常收数据；另外FIN报文段即使不携带数据也要占据一个序列号。
第二次挥手：服务端收到客户端发的FIN报文后给客户端回复确认报文，确认报文包含ACK标志位(ACK=1)、确认号ack=1102(客户端FIN报文序列号1101+1)、序列号seq=2300(300+2000)。此时服务端处于关闭等待状态，而不是立马给客户端发FIN报文，这个状态还要持续一段时间，因为服务端可能还有数据没发完。
第三次挥手：服务端将最后数据(比如50个字节)发送完毕后就向客户端发出连接释放报文，报文包含FIN和ACK标志位(FIN=1,ACK=1)、确认号和第二次挥手一样ack=1102、序列号seq=2350(2300+50)。
第四次挥手：客户端收到服务端发的FIN报文后，向服务端发出确认报文，确认报文包含ACK标志位(ACK=1)、确认号ack=2351、序列号seq=1102。注意客户端发出确认报文后不是立马释放TCP连接，而是要经过2MSL(最长报文段寿命的2倍时长)后才释放TCP连接。而服务端一旦收到客户端发出的确认报文就会立马释放TCP连接，所以服务端结束TCP连接的时间要比客户端早一些。

get请求和post请求的区别是什么？

1. Get是不安全的，因为在传输过程，数据被放在请求的URL中；Post的所有操作对用户来说都是不可见的。 但是这种做法也不时绝对的，大部分人的做法也是按照上面的说法来的，但是也可以在get请求加上 request body，给 post请求带上 URL 参数。

3. Get限制Form表单的数据集的值必须为ASCII字符；而Post支持整个ISO10646字符集。
4. Get执行效率却比Post方法好。Get是form提交的默认方法。
5. GET产生一个TCP数据包；POST产生两个TCP数据包。对于GET方式的请求，浏览器会把http header和data一并发送出去，服务器响应200（返回数据）；而对于POST，浏览器先发送header，服务器响应100 continue，浏览器再发送data，服务器响应200 ok（返回数据）。

### HTTP与HTTPS

```
http报文组成有那几部分？

一个HTTP请求报文由请求行（request line）、请求头部（header）、空行和请求数据4个部分组成

1.请求行
请求行由请求方法字段、URL字段和HTTP协议版本字段3个字段组成，它们用空格分隔。例如，GET /index.html HTTP/1.1。
HTTP协议的请求方法有GET、POST、HEAD、PUT、DELETE、OPTIONS、TRACE、CONNECT。

2.请求头部
请求头部由关键字/值对组成，每行一对，关键字和值用英文冒号“:”分隔。请求头部通知服务器有关于客户端请求的信息，典型的请求头有：
User-Agent：产生请求的浏览器类型。
Accept：客户端可识别的内容类型列表。
Host：请求的主机名，允许多个域名同处一个IP地址，即虚拟主机。

3.空行
最后一个请求头之后是一个空行，发送回车符和换行符，通知服务器以下不再有请求头。

4.请求数据
请求数据不在GET方法中使用，而是在POST方法中使用。POST方法适用于需要客户填写表单的场合。与请求数据相关的最常使用的请求头是Content-Type和Content-Length。

### 网址解析

```
输入一个网址后，解析过程是怎样的？

1：首先是查找浏览器缓存，浏览器会保存一段时间你之前访问过的一些网址的DNS信息，不同浏览器保存的时常不等。
2：如果没有找到对应的记录，这个时候浏览器会尝试调用系统缓存来继续查找这个网址的对应DNS信息。
3：如果还是没找到对应的IP，那么接着会发送一个请求到路由器上，然后路由器在自己的路由器缓存上查找记录，路由器一般也存有DNS信息。
4：如果还是没有，这个请求就会被发送到ISP（注：Internet Service Provider，互联网服务提供商，就是那些拉网线到你家里的运营商，中国电信中国移动什么的）
5：如果还是没有的话， 你的ISP的DNS服务器会将请求发向根域名服务器进行搜索。
6：浏览器终于得到了IP以后，浏览器接着给这个IP的服务器发送了一个http请求，方式为get
7：接收请求响应，渲染页面

### TCP和UDP

```
当http请求数据体过大时，TCP是怎么处理和分割报文的？

当 HTTP 请求的数据体过大时，TCP 会通过分段、流量控制和拥塞控制等机制来处理数据的传输。应用程序不需要关心底层 TCP 的分割和重组过程，TCP 会确保数据的可靠性和顺序性。

流量控制：TCP 是一种面向连接的协议，提供可靠的、顺序的数据传输。它使用流量控制机制（如滑动窗口）来确保发送方不会淹没接收方。
数据分段：TCP 将应用层数据分割成适合网络传输的段（segment），每个段都有自己的 TCP 头部。

当 HTTP 请求的数据体过大时，TCP 会执行以下步骤：
1：应用层数据准备：当应用程序（如 APM 系统的 HTTP 客户端）准备发送 HTTP 请求时，它会将请求数据体（如 JSON、XML 或其他格式）封装在 HTTP 消息中。

2：TCP分段：TCP 将应用层的数据分割成适合于网络传输的小段。这个过程称为“分段”。
每个 TCP 段的大小受多个因素影响，包括：
最大传输单元（MTU）：网络层的最大传输单元，通常为 1500 字节（以太网的标准 MTU）。TCP 段的大小通常会小于或等于 MTU 减去 TCP 和 IP 头部的大小。
TCP 窗口大小：流量控制机制，决定了在未收到确认的情况下可以发送的最大数据量。

3：数据发送：TCP 将这些段依次发送到网络中。每个段都会被封装在一个 IP 数据包中并通过底层网络传输。

4：接收和重组：接收方的 TCP 协议栈会接收这些 TCP 段，并根据序列号将它们重组为完整的应用层数据。如果某个段丢失，TCP 会通过重传机制请求重发丢失的段。

TCP 还实现了拥塞控制机制，以避免网络拥塞。在发送大数据体时，TCP 会根据网络的当前状况动态调整发送速率。这包括：
慢启动：初始发送速率较低，逐步增加。
拥塞避免：在网络出现拥塞时，减少发送速率。
快速重传：在丢失段的情况下，快速重发丢失的数据段。

如果使用 HTTP/2 协议，数据传输会更加高效。HTTP/2 通过将数据分成多个小帧（frame）进行传输，这样可以在同一 TCP 连接上并行处理多个请求和响应，减少延迟和提高效率。
```

```
聊一下TCP协议和UDP协议，dubbo底层是用的什么协议，优势是什么？GRPC底层是什么协议，优势是什么？

TCP（传输控制协议）
连接导向：TCP 是一种面向连接的协议，通信前需要建立连接（通过三次握手）。
可靠性：TCP 提供可靠的数据传输，确保数据包按顺序到达，并且可以进行重传以处理丢失的数据包。
流量控制：使用滑动窗口机制控制数据流量，避免接收方被淹没。
拥塞控制：TCP 实现了拥塞控制机制，动态调整数据发送速率以适应网络状况。
适用场景：适合需要高可靠性和顺序性的应用，如网页浏览、文件传输等。

UDP（用户数据报协议）
无连接：UDP 是一种无连接的协议，数据包可以在没有建立连接的情况下直接发送。
不可靠性：UDP 不保证数据包的送达、顺序或完整性，丢包、重复和乱序都是可能的
低延迟：由于没有建立连接和确认机制，UDP 通常比 TCP 更快，适合实时应用。
适用场景：适合对延迟敏感的应用，如视频流、语音通话和在线游戏等。

Dubbo 底层协议
Dubbo 默认使用的是 RPC（远程过程调用）协议，具体实现上通常是基于 TCP 协议。
高效：Dubbo 通过序列化和压缩技术，减少了数据传输的大小，提高了性能。
服务治理：提供丰富的服务治理功能，如负载均衡、服务降级、路由控制等。
扩展性：支持多种协议（如 HTTP、WebSocket、gRPC 等），可以根据需求选择不同的传输协议。
异步调用：支持异步和同步调用，适应不同的使用场景。

gRPC 底层协议
gRPC 默认使用 HTTP/2 协议作为传输层协议，这使得它能够利用 HTTP/2 的特性。
高效的序列化：gRPC 使用 Protocol Buffers 作为默认的序列化机制，相比 JSON 等格式更高效。
双向流：支持双向流通信，客户端和服务器可以同时发送和接收数据，适合实时应用。
多语言支持：gRPC 支持多种编程语言，方便不同平台和语言之间的互操作。
负载均衡和安全性：内置支持负载均衡、身份验证和加密，增强了服务的安全性和可靠性。

### DNS

```
讲一下DNS的解析过程

DNS（域名系统）的解析过程是将域名转换为 IP 地址的过程
1：用户请求：当用户在浏览器中输入一个域名（例如 www.example.com）并按下回车时，浏览器首先检查本地 DNS 缓存，看看是否已经存储了该域名的 IP 地址。如果有，直接使用该地址进行访问。

2：本地DNS解析器：如果本地缓存中没有该域名的记录，浏览器会向本地 DNS 解析器（通常是用户 ISP 提供的 DNS 服务器）发送 DNS 查询请求。这个解析器负责处理 DNS 查询并缓存结果。

3：根DNS服务器：如果本地 DNS 解析器无法解析该域名，它会向根 DNS 服务器发送查询。根 DNS 服务器是整个 DNS 系统的顶层，负责指向各个顶级域（如 .com, .org, .net 等）的 DNS 服务器。
如果本地 DNS 解析器无法解析该域名，它会向根 DNS 服务器发送查询。根 DNS 服务器是整个 DNS 系统的顶层，负责指向各个顶级域（如 .com, .org, .net 等）的 DNS 服务器。

4：顶级域DNS服务器：本地 DNS 解析器接收到根 DNS 服务器的响应后，接下来向顶级域 DNS 服务器发送查询请求。
顶级域 DNS 服务器会返回一个指向该域名的权威 DNS 服务器的地址。例如，对于 www.example.com，它会返回 example.com 的权威 DNS 服务器地址。

5：权威DNS服务器
本地 DNS 解析器接收到顶级域 DNS 服务器的响应后，接下来向权威 DNS 服务器发送查询请求。
权威 DNS 服务器会返回该域名的实际 IP 地址（例如，192.0.2.1），或者返回对应的其他记录（如 CNAME、MX 等）。

6：缓存结果：一旦本地 DNS 解析器获得了域名的 IP 地址，它会将该地址缓存一段时间（根据 DNS 记录的 TTL，生存时间）以便后续请求时可以直接使用，而不需要重新进行整个解析过程。

7：返回结果：最后，本地 DNS 解析器将 IP 地址返回给用户的浏览器，浏览器使用这个 IP 地址与目标服务器建立连接，完成网页的加载。

8：连接到目标服务器：浏览器使用得到的 IP 地址发送 HTTP 请求，连接到目标服务器，获取所需的网页内容。
```
